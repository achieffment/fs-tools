# examples — тестовое дерево для нормализатора

Песочница для ручного прогона `fs-normalizer`. Папки `01`–`08` соответствуют отдельным правилам из [`normalizer/rules.py`](../normalizer/rules.py), а `09-mix` — смешанным кейсам, где несколько правил срабатывают сразу. Внутри каждой — подпапка на кейс, а в ней несколько разнообразных примеров.

Каждый файл-пример продублирован в двух вариантах, чтобы проверить, что расширение не меняется, а имя без расширения обрабатывается целиком:

- `<имя>.txt`
- `<имя>` (без расширения)

Каталоги-примеры (для `CaseRule` и комбинированного пайплайна) создаются как настоящие директории с файлом-маркером `marker.txt` внутри — расширения у них нет по определению.

## Как пользоваться

Прогон рекурсивный и переименовывает объекты **на месте**. Так как `examples/` лежит под git, удобнее всего запускать прямо здесь: посмотреть изменения через git, а затем откатить одной командой.

### 1. Запуск

```bash
# из корня проекта; нормализатор спросит каталог интерактивно
./normalize.sh        # Linux/macOS (терминал)
./normalize.command   # macOS (двойной клик в Finder)
./normalize.bat       # Windows
# и укажите папку examples (или конкретную подпапку правила)
```

### 2. Просмотр изменений

```bash
git status                                    # переименованные/новые/удалённые объекты
git add -A examples
git diff --cached --stat -M -- examples       # переезды показаны как renamed
```

### 3. Откат к исходному дереву

```bash
examples/reset.sh        # Linux/macOS (терминал)
examples/reset.command   # macOS (двойной клик в Finder)
examples/reset.bat       # Windows
```

Скрипт без запроса возвращает отслеживаемые файлы к версии из git (`git restore`/`git checkout`) **и** удаляет неотслеживаемые объекты (например, опустевшие нормализованные каталоги — `git clean -fd`). Зачем второй шаг — см. ниже.

**Гарантии безопасности.** Очистка и восстановление выполняются автоматически, но безопасно:

- скрипт работает **только внутри `examples/`** (каталога, где лежит сам скрипт) — все операции ограничены pathspec `.`, поэтому ничего вне `examples/` не затрагивается ни при каких условиях;
- сами `reset.*` тоже приводятся к версии из git наравне с остальными файлами (в закоммиченном виде это ничего не меняет — содержимое совпадает); `.sh`/`.command` обёрнуты в функцию `main`, поэтому интерпретатор читает скрипт целиком до выполнения и корректно отрабатывает, даже если файл будет переписан на диске;
- отслеживаемые файлы лишь возвращаются к закоммиченной версии — это содержимое всегда восстановимо из git.

Поведение соответствует прямому `git checkout -- examples` + `git clean -fd examples`: всё в `examples/` приводится к состоянию из git. Это и есть смысл отката, поэтому учтите: незакоммиченные ручные правки фикстур и неотслеживаемые файлы внутри `examples/` будут отброшены (если такие правки нужно сохранить — закоммитьте их до запуска reset).

Скрытые объекты (имена на `.`) пропускаются, корневой каталог не переименовывается, повторный прогон ничего не меняет (идемпотентность).

> Намеренное исключение: в `08-edge-cases/conflict/` лежат `a b.txt` и `a-b.txt`. Первый нормализуется в имя второго, которое уже занято, поэтому нормализатор сообщает `Пропуск (конфликт)` и оба файла сохраняются — это ожидаемая демонстрация безопасной обработки конфликтов, а не ошибка.

### Почему `git status` не показывает часть каталогов

Git отслеживает только файлы, а не каталоги: для пустого каталога в git нет объекта, поэтому `git status` его не видит. Когда нормализатор переименовал, например, `dir-phrase/отчёт за март` в `dir-phrase/Otchiot-za-mart`, файл `marker.txt` переехал вместе с каталогом. Если затем сделать только `git checkout -- examples`, git вернёт `marker.txt` в исходный путь, но **опустевший** каталог `Otchiot-za-mart` останется как неотслеживаемый — git его «не замечает», а на следующем прогоне это имя уже занято и вызывает `Пропуск (конфликт)`.

Поэтому полный откат — это `git checkout` **плюс** `git clean -fd` (именно это и делает `reset.*`). Если же нужно, чтобы git отслеживал намеренно пустой каталог, в него кладут файл-заглушку (по соглашению `.gitkeep`); фикстурам это не требуется — в их листовых папках всегда есть файлы.

### Альтернатива: работа на копии

Вне git-репозитория (или чтобы не трогать рабочее дерево) можно прогонять на копии:

```bash
cp -r examples /tmp/examples-test
```

> Имена примеров намеренно не различаются только регистром. На регистронезависимой ФС (обычный Windows, macOS по умолчанию) файлы вроде `имя.txt` и `имя.TXT` схлопываются в один физический файл, из-за чего `git` показывает бесконечные «ложные» изменения после `pull`. Поэтому регистр расширения здесь не варьируется (то, что нормализатор сохраняет расширение и его регистр, проверяется юнит-тестами в [`tests/`](../tests)).

> Имена примеров сделаны переносимыми: без запрещённых на Windows символов (`< > : " | ? *`) и без пробела/точки в конце имени, иначе `git pull`/`checkout` на Windows падает с `invalid path`. Ведущие пробелы (например, `  отчёт`) допустимы и используются намеренно — это входной кейс для обрезки кромок.

## Ожидаемые результаты (оракул)

### 01-transliteration — `TransliterationRule`

Любой не-ASCII символ → ASCII (для файлов результат дополнительно в нижнем регистре).

| Подпапка | Вход | Результат (файл) |
|----------|------|------------------|
| cyrillic | `Отчёт` | `otchiot` |
| cyrillic | `Документ` | `dokument` |
| cyrillic | `Письмо` | `pismo` (мягкий знак удаляется, без апострофа) |
| accents | `café` | `cafe` |
| accents | `naïve` | `naive` |
| accents | `résumé` | `resume` |
| accents | `Müller` | `muller` |
| accents | `jamón` | `jamon` |
| greek | `Ωμέγα` | `omega` |
| greek | `αβγ` | `abg` |
| emoji-symbols | `hello🎉` | `hello` |
| emoji-symbols | `★star` | `star` |
| emoji-symbols | `привет👍` | `privet` |
| fractions | `½` | `01-02` (дробь `½`→`1/2`; `/` обезврежен в `-`, затем ведущие нули) |
| fractions | `10½` | `10-01-02` |
| quotes | `ООО «Печоралифтсервис»` | `ooo-pechoraliftservis` (`«»`→`<<`/`>>` — запрещены на Windows, вырезаются) |

Транслитерация может породить разделитель пути (`½`→`1/2`, `∖`/стрелки→`\`) или управляющий символ (U+2028/U+2029→перевод строки). Такие символы заменяются на `-`, чтобы имя всегда оставалось одним компонентом пути и объект нельзя было случайно переместить/потерять.

Кроме того, мягкий и твёрдый знаки (`ь`/`ъ`), которые unidecode превращает в апостроф, удаляются, а запрещённые в именах файлов Windows символы (`< > : " | ? *`, в которые превращается типографика вроде `«»`) вырезаются — иначе одиночный `<` в середине имени ломает переименование на Windows (`WinError 123`).

### 08-edge-cases — особые сценарии ФС

| Подпапка | Что проверяет | Ожидаемое поведение |
|----------|---------------|---------------------|
| hidden | скрытые объекты `.env`, `.secret/` | пропускаются (не переименовываются, внутрь не заходим); видимый сосед нормализуется |
| conflict | `a b.txt` + `a-b.txt` | `a b.txt` → `a-b.txt` конфликтует с существующим → пропуск, оба файла сохраняются |
| empty-stem | `@@@` | имя после чистки пустеет → возвращается без изменений |

| Вход (каталог) | Результат |
|----------------|-----------|
| `Отчёт 20.05.2020` | `Otchiot_2020-05-20` |

> `*` — результат **первого** прогона. Для этих строк нормализация **не идемпотентна**: одиночная цифра перед датой после добавления ведущего нуля превращается в `0X_YYYY`, и на втором прогоне `DateRule` распознаёт это как месяц-год, меняя имя дальше (например, `05_2020-05-20` → `2020-05-00_05-20`). Это особенность самого нормализатора, а не фикстур: миксы из `09-mix` намеренно подобраны так, чтобы её продемонстрировать. На обычных (нессостязательных) именах из папок `01`–`08` повторный прогон ничего не меняет.
