# examples — тестовое дерево для нормализатора

Песочница для ручного прогона нормализатора (`fs-normalizer`). Папки `01`–`06` и `09` соответствуют отдельным правилам из [`fs_tools/normalizer/rules/`](../../src/fs_tools/normalizer/rules/), `07-pipeline` — полному конвейеру, `08-edge-cases` — особым сценариям ФС, `10-ignore` демонстрирует фильтр путей [`.fs-nrm`](.fs-nrm) (он не правило имени, а отбор объектов до нормализации), а `11-mix` — смешанным кейсам, где несколько правил срабатывают сразу. Внутри каждой — подпапка на кейс, а в ней несколько разнообразных примеров.

Каждый файл-пример продублирован в двух вариантах, чтобы проверить, что расширение не меняется, а имя без расширения обрабатывается целиком:

- `<имя>.txt`
- `<имя>` (без расширения)

Каталоги-примеры (для `CaseRule` и комбинированного пайплайна) создаются как настоящие директории с файлом-маркером `marker.txt` внутри — расширения у них нет по определению.

Исключение — кейсы фильтра в `10-ignore`: они демонстрируют сопоставление путей, а не нормализацию имени, поэтому фикстуры повторяют реальные цели паттернов (в т.ч. настоящие расширения вроде `.bak`), и пара «без/с `.txt`» там необязательна.

## Как пользоваться

Канонический прогон — `--dry-run`: он показывает план без переименования объектов и
дописывает этот план в `.fs-log` с меткой `Режим: dry-run`. Production-прогон
рекурсивный и переименовывает объекты **на месте**.
Так как `examples/normalizer/` лежит под git, удобнее всего запускать прямо здесь:
посмотреть изменения через git, а затем откатить одной командой.

### 1. Запуск

```bash
# установленная команда; нормализатор спросит каталог интерактивно
fs-normalizer
# и укажите папку examples/normalizer (или конкретную подпапку правила)

# либо сразу аргументом, без диалога (минует интерактивный выбор):
fs-normalizer examples/normalizer

# канонический просмотр плана без изменений:
fs-normalizer examples/normalizer --dry-run

# то же через обёртки bin/ (создают .venv и editable-установку при первом запуске):
bin/normalize.sh         # Linux/macOS (терминал)
bin/normalize.command    # macOS (двойной клик в Finder)
bin/normalize.bat        # Windows
```

### 2. Просмотр изменений

```bash
git status                                 # переименованные/новые/удалённые объекты
git add -A examples
git diff --cached --stat -M -- examples    # переезды показаны как renamed
```

### 3. Откат к исходному дереву

```bash
examples/normalizer/reset.sh         # Linux/macOS (терминал)
examples/normalizer/reset.command    # macOS (двойной клик в Finder)
examples/normalizer/reset.bat        # Windows
```

Скрипт без запроса откатывает дерево к версии из git в три шага (`git reset` → `git clean -fdx` → `git checkout`), все — с `core.ignorecase=false`. Такой порядок и флаги нужны, чтобы корректно вернуть и обычные переименования, и те, что отличаются **только регистром**. Зачем именно так — см. ниже. Флаг `-x` (удаление в т.ч. игнорируемых файлов) заодно убирает журнал `.fs-log`; явный `rm`/`del` между `clean` и `checkout` оставлен как страховка.

**Гарантии безопасности.** Очистка и восстановление выполняются автоматически, но безопасно:

- скрипт работает **только внутри `examples/normalizer/`** (каталога, где лежит сам скрипт) — все операции ограничены pathspec `.`, поэтому ничего вне `examples/normalizer/` не затрагивается ни при каких условиях;
- сами `reset.*` тоже приводятся к версии из git наравне с остальными файлами (в закоммиченном виде это ничего не меняет — содержимое совпадает); `.sh`/`.command` обёрнуты в функцию `main`, поэтому интерпретатор читает скрипт целиком до выполнения и корректно отрабатывает, даже если файл будет переписан на диске;
- отслеживаемые файлы лишь возвращаются к закоммиченной версии — это содержимое всегда восстановимо из git.

Поведение соответствует прямому откату всего `examples/normalizer/` к состоянию из git. Это и есть смысл отката, поэтому учтите: незакоммиченные ручные правки фикстур и неотслеживаемые файлы внутри `examples/normalizer/` будут отброшены (если такие правки нужно сохранить — закоммитьте их до запуска reset).

Скрытые объекты (имена на `.`) пропускаются, корневой каталог не переименовывается, повторный прогон ничего не меняет (идемпотентность).

> В `examples/normalizer/` лежит закоммиченный демо-файл [`examples/normalizer/.fs-nrm`](.fs-nrm) — он читается из самого нормализуемого каталога (`examples/normalizer/`) и используется в секции «[10-ignore](#10-ignore--фильтр-путей-fs-nrm)». Все его паттерны привязаны к `10-ignore`, поэтому результаты в таблицах `01`–`09` и `11-mix` **не меняются** независимо от наличия файла. Проверки идемпотентности и конфликтов (ниже и в правилах) фильтр **снимают** (`rm -f .fs-nrm`), чтобы нормализовать всё дерево. Регистр в паттернах **не значим**: `Archive` и `archive` совпадают (как git `core.ignorecase=true`).

> Намеренное исключение: в `08-edge-cases/conflict/` лежат `a b.txt` и `a-b.txt`. Первый нормализуется в имя второго, которое уже занято, поэтому нормализатор сообщает `Пропуск (конфликт)` и оба файла сохраняются — это ожидаемая демонстрация безопасной обработки конфликтов, а не ошибка.

### Почему откат — это `reset` + `clean` + `checkout`, а не один `checkout`

Git отслеживает только файлы, а не каталоги: для пустого каталога в git нет объекта, поэтому `git status` его не видит. Когда нормализатор переименовал, например, `dir-phrase/отчёт за март` в `dir-phrase/Otchiot-za-mart`, файл `marker.txt` переехал вместе с каталогом. Если сделать только `git checkout -- examples`, git вернёт `marker.txt` в исходный путь, но **опустевший** каталог `Otchiot-za-mart` останется как неотслеживаемый — git его «не замечает», а на следующем прогоне это имя уже занято и вызывает `Пропуск (конфликт)`. Поэтому нужен ещё и `git clean -fd`, удаляющий неотслеживаемые объекты.

### Почему `core.ignorecase=false` и порядок `clean` → `checkout`

`CaseRule` капитализирует каталоги, поэтому многие объекты переименовываются **только по регистру** (`archive` → `Archive`, `accents` → `Accents`, файлы `MixedName` → `mixedname`). На **регистронезависимой ФС** (Windows, macOS по умолчанию; `core.ignorecase=true`) для git `Archive` и `archive` — один и тот же путь: обычный `git checkout`/`git restore` запишет содержимое в уже существующую запись, но **не переименует** её регистр, а `git clean` не тронет (считает отслеживаемой). В итоге папка так и останется `Archive` — откат «не сработает».

Поэтому `reset.*` выполняет все git-команды с `git -c core.ignorecase=false` и в порядке (удаление журнала `.fs-log` идёт между `clean` и `checkout`, см. выше):

    git -c core.ignorecase=false reset -q -- .        # снять индекс к HEAD: мис-кейс становится untracked
    git -c core.ignorecase=false clean -fdx -- .      # удалить untracked (мис-кейс + опустевшие каталоги + игнорируемые)
    git -c core.ignorecase=false checkout -- .        # пересоздать объекты с правильным регистром

С `ignorecase=false` мис-кейсовый объект считается неотслеживаемым, физически удаляется на шаге `clean` и пересоздаётся на `checkout` уже с верным регистром. Шаг `reset` снимает индекс к `HEAD` — на случай, если переименования были добавлены через `git add`. На регистрозависимой ФС (Linux) поведение не меняется.

Флаг `-x` у `clean` обязателен из-за демо-файла `08-edge-cases/hidden/.env`: в `.gitignore` он исключён шаблоном `.env`/`*.env`, а возвращён под версионный контроль негейтом, привязанным к **строчному** пути `…/hidden/.env`. После case-only переименования `hidden` → `Hidden` (`CaseRule`) негейт `…/Hidden/.env` уже не срабатывает, и `.env` снова считается игнорируемым. Без `-x` `git clean -fd` его не удаляет, мис-кейсовый каталог остаётся непустым, и на регистронезависимой ФС git не может его удалить — выдаёт интерактивное `Unlink of file '…/hidden' failed. Should I try again?` и зависает. С `-x` игнорируемый `Hidden/.env` удаляется, каталог исчезает, а `checkout` восстанавливает `hidden/` с верным регистром.

Если же нужно, чтобы git отслеживал намеренно пустой каталог, в него кладут файл-заглушку (по соглашению `.gitkeep`); фикстурам это не требуется — в их листовых папках всегда есть файлы.

### Альтернатива: работа на копии

Вне git-репозитория (или чтобы не трогать рабочее дерево) можно прогонять на копии:

```bash
cp -r examples /tmp/examples-test
```

> Имена примеров намеренно не различаются только регистром. На регистронезависимой ФС (обычный Windows, macOS по умолчанию) файлы вроде `имя.txt` и `имя.TXT` схлопываются в один физический файл, из-за чего `git` показывает бесконечные «ложные» изменения после `pull`. Поэтому регистр расширения здесь не варьируется (то, что нормализатор сохраняет расширение и его регистр, проверяется юнит-тестами в [`tests/`](../../tests)).

> Имена примеров сделаны переносимыми: без запрещённых на Windows символов (`< > : " | ? *`) и без пробела/точки в конце имени, иначе `git pull`/`checkout` на Windows падает с `invalid path`. Ведущие пробелы в git-tracked fixtures не используются (на Windows это даёт ложные `deleted/untracked`); сценарий обрезки кромок проверяется юнит-тестами (`tests/normalizer/test_name.py`).

### 01-transliteration — `TransliterationRule`

Любой не-ASCII символ → ASCII (для файлов результат дополнительно в нижнем регистре).

| Подпапка      | Вход                     | Результат (файл)                                                            |
|---------------|--------------------------|-----------------------------------------------------------------------------|
| cyrillic      | `Отчёт`                  | `otchiot`                                                                   |
| cyrillic      | `Документ`               | `dokument`                                                                  |
| cyrillic      | `Письмо`                 | `pismo` (мягкий знак удаляется, без апострофа)                              |
| accents       | `café`                   | `cafe`                                                                      |
| accents       | `naïve`                  | `naive`                                                                     |
| accents       | `résumé`                 | `resume`                                                                    |
| accents       | `Müller`                 | `muller`                                                                    |
| accents       | `jamón`                  | `jamon`                                                                     |
| greek         | `Ωμέγα`                  | `omega`                                                                     |
| greek         | `αβγ`                    | `abg`                                                                       |
| emoji-symbols | `hello🎉`                 | `hello`                                                                     |
| emoji-symbols | `★star`                  | `star`                                                                      |
| emoji-symbols | `привет👍`                | `privet`                                                                    |
| fractions     | `½`                      | `01-02` (дробь `½`→`1/2`; `/` обезврежен в `-`, затем ведущие нули)         |
| fractions     | `10½`                    | `10-01-02`                                                                  |
| quotes        | `ООО «Печоралифтсервис»` | `ooo-pechoraliftservis` (`«»`→`<<`/`>>` — запрещены на Windows, вырезаются) |

Транслитерация может породить разделитель пути (`½`→`1/2`, `∖`/стрелки→`\`) или управляющий символ (U+2028/U+2029→перевод строки). Такие символы заменяются на `-`, чтобы имя всегда оставалось одним компонентом пути и объект нельзя было случайно переместить/потерять.

Кроме того, мягкий и твёрдый знаки (`ь`/`ъ`), которые unidecode превращает в апостроф, удаляются, а запрещённые в именах файлов Windows символы (`< > : " | ? *`, в которые превращается типографика вроде `«»`) вырезаются — иначе одиночный `<` в середине имени ломает переименование на Windows (`WinError 123`).

### 05-trim-edge — `TrimEdgeRule`

Обрезка не буквенно-цифровых символов по краям (ведущий `_` сохраняется и у файлов, и у папок). Исключение — `+` и `#`: это символы имени, они сохраняются по обоим краям (`C#`, `C++`, `F#`, `notepad++`), иначе хвостовые `#`/`++` срезались бы и `C#`/`C++` схлопывались бы в `C`.

| Подпапка           | Вход         | Результат (файл)                                 |
|--------------------|--------------|--------------------------------------------------|
| edges              | `-file-`     | `file`                                           |
| edges              | `--name--`   | `name`                                           |
| leading-underscore | `_private`   | `_private`                                       |
| leading-underscore | `_config`    | `_config`                                        |
| dunder             | `__name__`   | `__name`                                         |
| balanced-paren     | `(text)-end` | `(text)-end` (парная скобка на краю сохраняется) |
| balanced-paren     | `[text]-end` | `[text]-end` (то же для квадратной скобки)       |
| keep-plus          | `C++`        | `c++` (`+` сохраняется по краю)                  |
| keep-hash          | `C#`         | `c#` (`#` сохраняется по краю)                   |

Папки — с заглавной буквы, файлы — в нижнем регистре. Исключение — общепринятые имена вроде `README`: их регистр сохраняется. У папок ведущий `_` сохраняется, а капитализируется первая буква уже после него (`_private` → `_Private`).

### 08-edge-cases — особые сценарии ФС

| Подпапка   | Что проверяет                      | Ожидаемое поведение                                                                 |
|------------|------------------------------------|-------------------------------------------------------------------------------------|
| hidden     | скрытые объекты `.env`, `.secret/` | пропускаются (не переименовываются, внутрь не заходим); видимый сосед нормализуется |

| empty-stem | `@@@`                              | имя после чистки пустеет → возвращается без изменений                               |

### 09-brackets — `BracketsRule`

Содержимое круглых и квадратных скобок: число/дата (без ASCII-букв) → обе скобки убираются (а число дальше дополняется ведущим нулём `LeadingZeroRule`); текст → скобки сохраняются. Закрывающая скобка у текстового варианта не срезается `TrimEdgeRule` (см. сохранение парной скобки на краю). Непарные и несовпадающие скобки (`Файл (1`, `инн (текст]`) вырезаются как невалидные.

| Подпапка  | Вход                        | Результат (файл)                                     |
|-----------|-----------------------------|------------------------------------------------------|
| digit     | `Файл (1)`                  | `fail-01` (скобки убраны, ведущий ноль)              |
| digit     | `Файл (12)`                 | `fail-12` (скобки убраны, двузначное без нуля)       |
| digit     | `Файл [2]`                  | `fail-02` (квадратные скобки — так же)               |
| text      | `инн (Нового договора нет)` | `inn-(novogo-dogovora-net)` (скобки сохранены)       |
| text      | `инн [Нового договора нет]` | `inn-[novogo-dogovora-net]` (квадратные — так же)    |
| date      | `(2021.03.10)`              | `2021-03-10` (дата без букв → скобки убраны)         |
| unmatched | `Файл (1`                   | `fail-01` (непарная скобка вырезана, ведущий ноль)   |
| unmatched | `инн (Нового договора нет`  | `inn-novogo-dogovora-net` (непарная скобка вырезана) |
| unmatched | `Файл (2]`                  | `fail-02` (несовпадающие скобки вырезаны)            |

### 10-ignore — фильтр путей `.fs-nrm`

Демонстрация [`examples/normalizer/.fs-nrm`](.fs-nrm) (gitignore-семантика поверх `pathspec`): он лежит в `examples/normalizer/`, читается из неё, и якоря `/` отсчитываются от этого корня. Чтобы прочие примеры не затрагивались, **все** его паттерны привязаны к `10-ignore`. Прогон — на самой папке `examples/normalizer/` (фильтр читается из `examples/normalizer/.fs-nrm`).

| Подпапка         | Паттерн в `.fs-nrm`                                                                                                | Поведение                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|------------------|--------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| anchored         | `/10-ignore/anchored/archive`                                                                                      | каталог `archive` (с ведущим `/` — якорь от корня) и его содержимое не нормализуются; сосед `current` — нормализуется                                                                                                                                                                                                                                                                                                                                                                                                                    |
| relative         | `10-ignore/relative/archive`                                                                                       | то же: без ведущего `/`, но срединный `/` якорит паттерн от корня так же, как `/...`                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| reincluded       | `/10-ignore/reincluded` + `!/10-ignore/reincluded/proc`                                                            | каталог `reincluded` исключён целиком (имя и `Черновик письма` не тронуты), но подкаталог `proc` возвращён правилом-`!` и нормализуется                                                                                                                                                                                                                                                                                                                                                                                                  |
| basename         | `cache`                                                                                                            | паттерн без `/` (basename) исключает объект `cache` на любой глубине под `10-ignore`; сосед `Готовый материал` — нормализуется                                                                                                                                                                                                                                                                                                                                                                                                           |
| file-glob        | `/10-ignore/file-glob/*.bak` + `!/10-ignore/file-glob/Сохранить.bak`                                               | по расширению исключаются `.bak`-файлы (`Черновик.bak` не тронут), но `Сохранить.bak` возвращён правилом-`!` и нормализуется; не-`.bak` сосед `Заметка.txt` под паттерн не попадает                                                                                                                                                                                                                                                                                                                                                      |
| file-exact       | `/10-ignore/file-exact/Пароли.txt`                                                                                 | исключён ровно один файл по полному пути; расширение значимо — одноимённый `Пароли` (без `.txt`) под паттерн не попадает и нормализуется                                                                                                                                                                                                                                                                                                                                                                                                 |

| hash             | `/10-ignore/hash/C# заметки`                                                                                       | имя каталога содержит `#`: срединный `#` — литерал (комментарий только в начале строки), поэтому паттерн исключает папку; сосед `Сборка` — нормализуется                                                                                                                                                                                                                                                                                                                                                                                 |
| plus             | `/10-ignore/plus/C++ проекты`                                                                                      | имя каталога содержит `++`: `+` — обычный символ (не метасимвол), поэтому паттерн исключает папку; сосед `Прочее` — нормализуется                                                                                                                                                                                                                                                                                                                                                                                                        |
| nested-reinclude | `/10-ignore/nested-reinclude/Activities/*/Projects` + `!/10-ignore/nested-reinclude/Activities/*/Projects/**/Data` | проект занятия (`*` — имя занятия) исключён целиком, но папки `Data` возвращены правилом-`!` на любой глубине (`**` сквозь сегменты): нормализуется только содержимое `Data` — и под исключённым `Archive` (`Addl/Archive/example.com/Data/Выгрузка 2021` → `vygruzka_2021-00-00`), и у сайта прямо в `Addl` (`Addl/another.com/Data/Выгрузка 2022` → `vygruzka_2022-00-00`). Всё, что не `Data`, не тронуто: соседние `Back`, промежуточные каталоги и сайт `Addl/example.com` без `Data`; а `Resources` вне `Projects` — нормализуется |

Исключение объекта не «прикрепляет» его родителя: например, `anchored/` (имя кейса) капитализируется в `Anchored`, а исключённый `archive` внутри остаётся `archive` — это поведение gitignore (то же и с файловыми кейсами: каталог `File-glob` нормализуется, а исключённый `Черновик.bak` внутри — нет). Матчинг при этом **регистронезависим** (как git `core.ignorecase=true`), поэтому капитализированный родитель (`File-glob`) по-прежнему совпадает с якорем паттерна (`/10-ignore/file-glob/*.bak`): повторный прогон по `10-ignore` **ничего не меняет** даже при наличии `.fs-nrm` (фильтр идемпотентен между прогонами). Сам `.fs-nrm` скрыт (имя на `.`) и обходом пропускается, поэтому не нормализуется. Если файл снять, всё дерево `10-ignore` нормализуется обычным порядком (идемпотентно, без конфликтов).

> Нормализация **идемпотентна** для всех кейсов, включая эти миксы. Одиночная цифра-индекс перед датой получает ведущий ноль (`0X_YYYY-…`), но `_` не считается разделителем месяц-год, поэтому на повторном прогоне `DateRule` не принимает `0X_YYYY` за дату и готовая дата не ломается (`05_2020-05-20` остаётся `05_2020-05-20`). Поэтому второй прогон по всему дереву `examples/normalizer/` ничего не переименовывает; единственный `Пропуск (конфликт)` — задокументированный кейс `08-edge-cases/conflict`.
