# examples/schemer — тестовое дерево для проверки схемы базы знаний

Песочница для ручного прогона проверки схемы (`fs-schemer`). Проверяемый корень —
[`Warehouse/`](Warehouse/) (там же лежит `.fs-sch.toml` — конфиг читается из того же
каталога, что передан аргументом или выбран интерактивно). Этот `README.md` лежит
**вне** проверяемого корня, поэтому сам не попадает под F15.

Дерево воспроизводит один тематический узел `Warehouse/Code` со всеми четырьмя
группами `.fs-sch.toml` (`_Knowledges`, `_Commands`, `_Blueprints`,
`_Resources`) и содержит **намеренные** нарушения, покрывающие все категории F1–F15:

- `Code/_Commands/_main.md` отсутствует — обязательный служебный файл (F4,
  `missing_group_file`);
- `Code/_Blueprints/_devs.md` есть, но заголовок не совпадает — опциональный файл
  с контент-правилом (F7, `bad_header`);
- `Code/_Resources` не содержит ни одного видимого файла (даже вложенного) —
  пустая группа (F14, `empty_group`);
- `Code/loose.md` лежит напрямую в тематическом узле, минуя групповые папки (F15,
  `loose_file`).

Утилита **не меняет структуру** дерева (read-only), поэтому reset-скриптов здесь
нет. Единственная запись — журнал `.fs-log` в `Warehouse/` (скрыт, в репозиторий не
попадает через `.gitignore`, на отчёт не влияет).

`_Resources/.gitkeep` — заглушка для пустого каталога (git не хранит пустые папки);
имя скрыто (на `.`), поэтому не считается видимым файлом и не снимает пометку
«пусто» у F14.

## Как пользоваться

```bash
# установленная команда; утилита спросит каталог интерактивно
fs-schemer
# и укажите папку examples/schemer/Warehouse

# либо сразу аргументом, без диалога:
fs-schemer examples/schemer/Warehouse

# то же через обёртки bin/ (создают .venv и editable-установку при первом запуске):
bin/scheme.sh         # Linux/macOS (терминал)
bin/scheme.command    # macOS (двойной клик в Finder)
bin/scheme.bat        # Windows
```

## Файл конфигурации

[`Warehouse/.fs-sch.toml`](Warehouse/.fs-sch.toml) демонстрирует полный формат
(см. [`scheme-format.md`](../../.claude/rules/scheme-format.md)): четыре группы,
`[defaults].exclude_prefix = "_"`, единый механизм `[[group.file]]` для обязательных
(`_main.md`) и опциональных (`_commands.md`, `rules.md`, …, `_devs.md`) служебных
файлов, `default_rule` — контент-правило для «обычных» файлов группы.

## Ожидаемый результат

Терминал выводит только заголовок, статус и сводку — минимум информации:

```text
Каталог: examples/schemer/Warehouse
Статус: error. Найдены нарушения структуры/контента.
Сводка: проверено групп: 4; проверено файлов: 6; нарушений: 4.
```

Список нарушений (тип + путь, для контентных — ожидание/факт) уходит в `.fs-log`:

```text
заголовок не совпадает: Code/_Blueprints/_devs.md (ожидается «# Наработки», факт «# Плохой заголовок»)
отсутствует обязательный файл: Code/_Commands/_main.md
пустая группа: Code/_Resources
файл вне групповой папки: Code/loose.md
```

Как читать результат:

| Нарушение                   | Почему попало в отчёт                                                            |
|-----------------------------|----------------------------------------------------------------------------------|
| `Code/_Blueprints/_devs.md` | опциональный файл присутствует, но строка 1 — не «# Наработки»                   |
| `Code/_Commands/_main.md`   | обязательный служебный файл группы `_Commands` отсутствует                       |
| `Code/_Resources`           | группа существует, но ни одного видимого файла в ней (и вложенных подпапках) нет |
| `Code/loose.md`             | файл лежит прямо в тематическом узле `Code`, а не в одной из групповых папок     |

Чего в отчёте **нет** (и это правильно):

- `Code/_Knowledges/*` — `_main.md`, `note.md`, `rules.md` соответствуют своим
  контент-правилам (обязательному, `default_rule` и опциональному соответственно);
- `Code/_Commands/cmd.md` — обычный файл группы, соответствует `default_rule`
  (`# Команды`); отсутствие `_main.md` в той же папке не влияет на его проверку;
- `Code/_Blueprints/tpl.md` — обычный файл, соответствует `default_rule`
  (`# Шаблоны`);
- `Warehouse/.fs-sch.toml` — под F15 не попадает: он скрытый (ведущая точка),
  как и остальные конфиги-артефакты.

Чтобы увидеть «нарушений нет» (код возврата `0`), исправьте перечисленные пункты
вручную (например, добавьте `_Commands/_main.md` с текстом `# Команды`) и повторите
запуск — утилита ничего на диске не меняет сама.
