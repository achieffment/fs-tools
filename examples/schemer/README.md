# examples/schemer — тестовое дерево для проверки схемы базы знаний

Песочница для ручного прогона проверки схемы (`fs-schemer`). Проверяемый корень —
[`Warehouse/`](Warehouse/), а конфиг [`.fs-sch.toml`](.fs-sch.toml) лежит рядом с
этим `README.md`, **вне** `Warehouse/` — демонстрирует `[defaults].apply_root`
(см. [`scheme-format.md`](../../.claude/rules/scheme-format.md)): утилите передаётся
`examples/schemer` (там читается конфиг), а `apply_root = "Warehouse"`
перенаправляет обход в `Warehouse/`. Раз конфиг вне проверяемого корня, вопрос
F15 для него не встаёт вовсе (в отличие от прежней раскладки, где помогала только
скрытость точки).

Дерево воспроизводит один тематический узел `Warehouse/Code` со всеми пятью
группами `.fs-sch.toml` (`_Knowledges`, `_Commands`, `_Blueprints`,
`_Resources`, `_Archive`) и содержит **намеренные** нарушения, покрывающие все
категории F1–F15 плюс обе стороны `strict`:

- `Code/_Commands/_main.md` отсутствует — обязательный служебный файл (F4,
  `missing_group_file`);
- `Code/_Blueprints/_devs.md` есть, но заголовок не совпадает — опциональный файл
  с контент-правилом (F7, `bad_header`);
- `Code/_Archive` не содержит ни одного видимого файла (даже вложенного) —
  пустая группа (F14, `empty_group`);
- `Code/loose.md` лежит напрямую в тематическом узле, минуя групповые папки (F15,
  `loose_file`);
- `Code/_Commands` — `strict = true`: внутри лежит вложенный `Old/legacy.md`, и он
  тоже даёт `loose_file` — подпапки этой группы заново классифицируются наравне с
  остальным деревом (для групп, где вложенность — ошибка, а не норма);
- `Code/_Resources` — `strict` не задан (`false` по умолчанию): внутри лежит
  вложенная свалка `Library/Sub/asset.bin`, но нарушения нет — обход не спускается
  в подпапки непрозрачной по умолчанию группы, поэтому F15 на `Library`/`Sub` не
  срабатывает (без `strict = true` каждый файл в такой вложенной структуре давал
  бы отдельный `loose_file` — это и был исходный баг-репорт, из-за которого дефолт
  сменился).

Утилита **не меняет структуру** дерева (read-only), поэтому reset-скриптов здесь
нет. Единственная запись — журнал `.fs-log.log`, который пишется рядом с конфигом
(`examples/schemer/`, а не `Warehouse/`) — скрыт, в репозиторий не попадает через
`.gitignore`, на отчёт не влияет.

`_Archive/.gitkeep` — заглушка для пустого каталога (git не хранит пустые папки);
имя скрыто (на `.`), поэтому не считается видимым файлом и не снимает пометку
«пусто» у F14.

## Как пользоваться

```bash
# установленная команда; утилита спросит каталог интерактивно
fs-schemer
# и укажите папку examples/schemer — конфиг сам перенаправит проверку в Warehouse/

# либо сразу аргументом, без диалога:
fs-schemer examples/schemer

# то же через обёртки bin/ (создают .venv и editable-установку при первом запуске):
bin/scheme.sh         # Linux/macOS (терминал)
bin/scheme.command    # macOS (двойной клик в Finder)
bin/scheme.bat        # Windows
```

## Файл конфигурации

[`.fs-sch.toml`](.fs-sch.toml) демонстрирует полный формат (см.
[`scheme-format.md`](../../.claude/rules/scheme-format.md)): `[defaults].apply_root =
"Warehouse"` (конфиг и проверяемое дерево разнесены), `[defaults].exclude_prefix =
"_"`, пять групп, единый механизм `[[group.file]]` для обязательных (`_main.md`) и
опциональных (`_commands.md`, `rules.md`, …, `_devs.md`) служебных файлов,
`default_rule` — контент-правило для «обычных» файлов группы, `strict = true` у
`_Commands` — включает прежнее строгое поведение (вложенность в этой группе —
ошибка), тогда как `_Resources` оставлена без `strict` (дефолт `false` — вложенность
разрешена). `_Blueprints.default_rule` дополнительно задаёт `extensions = [".md"]`
(whitelist по расширению) — демонстрируется на `Code/_Blueprints/asset.bin`
(см. ниже).

## Ожидаемый результат

Терминал выводит только заголовок, статус и сводку — минимум информации:

```text
Каталог: examples/schemer/Warehouse
Статус: error. Найдены нарушения структуры/контента.
Сводка: проверено групп: 5; проверено файлов: 6; нарушений: 5.
```

Список нарушений (тип + путь, для контентных — ожидание/факт) уходит в `.fs-log.log`:

```text
заголовок не совпадает: Code/_Blueprints/_devs.md (ожидается «# Наработки», факт «# Плохой заголовок»)
файл вне групповой папки: Code/_Commands/Old/legacy.md
отсутствует обязательный файл: Code/_Commands/_main.md
пустая группа: Code/_Archive
файл вне групповой папки: Code/loose.md
```

Как читать результат:

| Нарушение                      | Почему попало в отчёт                                                            |
|--------------------------------|----------------------------------------------------------------------------------|
| `Code/_Blueprints/_devs.md`    | опциональный файл присутствует, но строка 1 — не «# Наработки»                   |
| `Code/_Commands/Old/legacy.md` | `_Commands` объявлена `strict = true` — подпапка `Old` заново классифицируется   |
| `Code/_Commands/_main.md`      | обязательный служебный файл группы `_Commands` отсутствует                       |
| `Code/_Archive`                | группа существует, но ни одного видимого файла в ней (и вложенных подпапках) нет |
| `Code/loose.md`                | файл лежит прямо в тематическом узле `Code`, а не в одной из групповых папок     |

Чего в отчёте **нет** (и это правильно):

- `Code/_Knowledges/*` — `_main.md`, `note.md`, `rules.md` соответствуют своим
  контент-правилам (обязательному, `default_rule` и опциональному соответственно);
- `Code/_Commands/cmd.md` — обычный файл группы, соответствует `default_rule`
  (`# Команды`); отсутствие `_main.md` в той же папке не влияет на его проверку;
- `Code/_Blueprints/tpl.md` — обычный файл, соответствует `default_rule`
  (`# Шаблоны`);
- `Code/_Resources/Library/Sub/asset.bin` — лежит на два уровня глубже прямых
  детей `_Resources`; `strict` не задан (`false` по умолчанию) — обход не
  спускается в подпапки этой группы, F15 сюда не заходит;
- `Code/_Blueprints/asset.bin` — бинарный файл (невалидный UTF-8) лежит прямо в
  группе, но `default_rule` этой группы задаёт `extensions = [".md"]` — файл не
  подходит под фильтр, поэтому вообще не читается: без фильтра тот же файл дал бы
  ложный `read_error` (именно этот сценарий и мотивировал появление
  `extensions`/`exclude_extensions`, см. [`scheme-format.md`](../../.claude/rules/scheme-format.md));
- `.fs-sch.toml` — вообще не участвует в обходе: он лежит вне проверяемого
  корня `Warehouse/` (в `examples/schemer/`, рядом с этим `README.md`).

Чтобы увидеть «нарушений нет» (код возврата `0`), исправьте перечисленные пункты
вручную (например, добавьте `_Commands/_main.md` с текстом `# Команды`) и повторите
запуск — утилита ничего на диске не меняет сама.
