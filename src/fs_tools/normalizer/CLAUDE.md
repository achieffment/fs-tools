# CLAUDE.md — `src/fs_tools/normalizer/`

Путь-локальные правила для режима `normalizer` (адаптировано из
`.cursor/rules/path-matching.mdc` — область `ignore.py`/`engine.py` — и
`.cursor/rules/date-rule.mdc` — область `rules/date.py`). Общие правила
проекта — см. `../../../CLAUDE.md`.

## Фильтр путей `.fs-ignore` (gitignore-семантика)

Фильтр (`ignore.py`) — настоящая `.gitignore`-семантика поверх `pathspec`.
Инварианты приняты намеренно — НЕ «чинить» без явной задачи.

**Движок и формат**: единый файл `.fs-ignore` лежит в нормализуемом каталоге
(корне `apply`) и читается из него (`load_fs_ignore(root)` в `runner.py`, НЕ
из корня проекта). Стиль `.gitignore`: обычная строка исключает, `!` —
возвращает (override). Паттерны строятся через
`pathspec.PathSpec.from_lines(_FACTORY, ...)`, где `_FACTORY` импортируется
из `fs_tools/shared/pathspec_compat.py` и выбирается из доступных
(`gitignore`, иначе `gitwildmatch`) — не привязывайся к версии pathspec.
Файл НИКОГДА не изменяется (только чтение, `utf-8-sig`). Нет файла → `None`
(фильтр выключен); пустой/из комментариев → `FsIgnore` без правил.
Комментарий — только ведущий `#`. Рабочий демо-файл —
`examples/normalizer/.fs-ignore` (секция `10-ignore`); его паттерны
привязаны к `10-ignore`. Под git отслеживается.

**Семантика матчинга (gitignore)**: `*` — символы в пределах сегмента; `**`
— ноль или более сегментов; `?` — один символ; `[abc]`/`[a-z]` — класс
символов; завершающий `/` (`build/`) — только каталог и содержимое.

- **Смена контракта**: `[abc]` и `?` — активные метасимволы. Литеральные
  скобки/`?` экранируются `\` (`Файл \[1\]`). Разделитель — только `/`; `\`
  — экранирование.
- **Префикс `./` не поддерживается** (`.` — обычный сегмент). Пиши `foo`
  (basename) или `/foo` (якорь).
- **`#` — только ведущий**: срединный `#` литерален (`C#`, `C++`).
- **Якоря**: паттерн без `/` (или только с завершающим `/`) — basename
  (любая глубина); паттерн с `/` внутри/в начале — привязан к корню
  нормализации.
- **Порядок строк значим**: решение принимает последняя совпавшая строка;
  `!` возвращает убранное ранее.
- **Регистронезависимость** (как git `core.ignorecase=true`): `Archive` ==
  `archive`. Паттерны пересобираются с `re.IGNORECASE`. Мотив: `CaseRule`
  капитализирует родителей (`file-glob` → `File-glob`), и регистрозависимый
  якорь рвал бы идемпотентность фильтра.

**Относительность и обход**: матчинг относительно корня `apply`:
`FsNormalizer` считает `path.relative_to(root)` и зовёт
`FsIgnore.matches(rel, is_dir)`; каталогам добавляется завершающий `/`.
`probe` (в `engine.py`) включается при `FsIgnore.has_overrides()` (есть
правило-`!`): исключённые каталоги обходятся (внутри возможны возвращённые
потомки). Без `!` исключённый каталог обрезается. Исключённые объекты не
попадают в счётчики `renamed`/`skipped`. **Скрытые (на `.`) — раньше
фильтра**: `_hidden` отсекает их ДО `.fs-ignore`, поэтому `!` их не
возвращает. Сам `.fs-ignore` скрыт — пропускается обходом.

**Осознанные допущения**: `!` может вернуть потомка внутри исключённого
каталога (оцениваем каждый путь независимо) — намеренно, под сценарий
«исключить каталог, вернуть отдельные файлы». Относительная gitignore-модель:
матчинг родителей выше выбранного каталога и токен диска (`D:` ↔ `/mnt/d`) не
поддерживаются — не возвращай без явной задачи. Регистронезависимость лечит
только разницу в регистре родителей. Не-ASCII имя родителя
`TransliterationRule` меняет целиком (`Документы` → `Dokumenty`) — такой якорь
рвётся на повторном прогоне (известное ограничение).

## `DateRule` (`rules/date.py`) — осознанные допущения

Решения ниже приняты намеренно и защищают главный инвариант —
идемпотентность. Это НЕ баги: не «чинить» без явной задачи (изменение тянет
за собой все четыре пласта).

- **Уже-ISO форма не валидируется по календарю.** Группа `norm`
  (`\d{4}-\d{2}-\d{2}`) распознаёт *форму* нормализованной даты и
  возвращает её без изменений — даже если по календарю она невалидна
  (`2020-13-45`). Это страж идемпотентности, а не валидатор. Не добавляй
  сюда календарную проверку.
- **`_` не разделитель месяц-год.** В группе `mmyy` (`MM.YYYY`/`YYYY.MM`)
  разделители — только `. - /`, но НЕ `_` (подчёркивание — собственный
  токен-разделитель нормализатора). Иначе индекс с ведущим нулём рядом с
  годом (`5 ` → `05_2020-…`) на повторном прогоне читался бы как `MM_YYYY` и
  ломал готовую дату (`05_2020-05-20` → `2020-05-00_05-20`). В трёхполевой
  `full` (`YYYY_MM_DD`) `_` оставлен: форма однозначна. Не возвращай `_` в
  `mmyy`.
- **Диапазон года зависит от структуры токена.** Голый 4-значный токен
  считается годом только в диапазоне 1900–2099 (чтобы `1080`/`12345` не
  стали датами). Когда разделители задают однозначную структуру
  (`DD.MM.YYYY`, `MM.YYYY`, `YYYY.MM`), год принимается в любом 4-значном
  значении (для полной даты — в пределах `datetime`). Асимметрия намеренная:
  структура снимает неоднозначность. Не «унифицируй» диапазон.
