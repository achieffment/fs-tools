# CLAUDE.md — `src/fs_tools/syncher/`

Путь-локальные правила для режима `syncher` (адаптировано из
`.cursor/rules/config-format.mdc`, `offload-safety.mdc`, `rsync-mapping.mdc`).
Общие правила проекта — см. `../../../CLAUDE.md`.

## Формат `.fs-sync.toml`

Конфиг лежит в корне синхронизируемого каталога (путь — позиционный аргумент
CLI). Разбор — стандартный `tomllib` (Python 3.11+). Модель — `config.py`.

Секции: `[defaults]` — значения по умолчанию для всех профилей; `[[sync]]` —
профили зеркалирования (ПК → сервер, с зеркалированием удалений); `[[backup]]`
— профили offload (выгрузка с последующим локальным удалением/архивом).

Поля профиля:

- `name` — уникальное имя (обязательно).
- `local_root` — путь относительно конфига или абсолютный (обязателен,
  должен существовать).
- `remote_root` — `user@host:/path`, `alias:/path` из `~/.ssh/config` или
  локальный путь (обязателен; локальный отсчитывается от каталога конфига).
- `exclude` / `include` — списки gitignore-подобных паттернов (см. ниже,
  «Трансляция в rsync»).
- `delete` — зеркалить удаления (дефолт: `true` для sync, `false` для
  backup).
- `dry_run` — дефолт `false`.
- `delete_threshold` (число, дефолт 100) и `delete_threshold_pct` (доля %,
  дефолт 25) — пороги delete-guard; `force_delete` (дефолт `false`) снимает
  защиту.
- Передача: `checksum`, `compress`, `partial_progress` (bool), `bwlimit`
  (строка), `ssh_opts` (список строк).
- Только `[[backup]]`: `after_push` (`delete`|`archive`|`nothing`, дефолт
  `nothing`), `verify` (bool, дефолт `true`), `archive_dir` (для `archive`;
  дефолт `<local_root>/../_fs-backup/<profile>/<YYYY-MM-DD>/`).

Валидация → `ConfigError` (код возврата 1): уникальность `name` среди всех
профилей; `local_root` существует; `remote_root` непустой и безопасный (не
корень `/`, непустой путь после `:`); `after_push` — корректный enum; типы
полей соответствуют ожидаемым. Сообщение об ошибке указывает профиль и поле.

Поле профиля перекрывает одноимённое из `[defaults]`. Меняя набор полей или
дефолты, синхронизируй `config.py`, тесты, README и `examples/syncher/`.

Legacy-ключи TOML — внешний контракт утилиты. Внутренняя модель может
использовать свои имена (`source_path`/`target_path`/`backup_path`) и
маппинг, но поведение `rsync` и других библиотек остаётся штатным.

## Offload: безопасность локального удаления

Профиль `[[backup]]` выгружает каталог на сервер и при `after_push` ∈
{`delete`, `archive`} убирает локальные файлы. Реализация — `offload.py`.
Главный инвариант: **локальный файл удаляется/переносится только после
подтверждённой передачи**; частичный успех не трогает непереданное.

Поток (`run_offload`):

1. Передать каталог на сервер (по умолчанию без `--delete` — накопительный
   архив).
2. Сбой передачи (`rsync` вернул ненулевой код) → код 2, `after_push` НЕ
   выполняется.
3. `--dry-run` → только план передачи; локальные файлы не трогаются.
4. При `verify = true` повторный `rsync --dry-run --checksum` даёт множество
   «ещё к передаче» (`verify_pending`). Подтверждены = файлы области профиля
   (`source_files`) минус это множество.
5. `after_push` применяется только к подтверждённым:
   - `delete` — удалить локальный файл, затем подчистить опустевшие
     каталоги (но не сам `local_root`); сохраняются только якорные
     include-каталоги (и уровни, явно заданные отдельным якорным
     include-паттерном); для одного якоря защищается ближайший уровень,
     вложенный одноимённый каталог не защищается автоматически;
   - `archive` — перенести в `archive_dir` (дефолт
     `<local_root>/../_fs-backup/<profile>/<YYYY-MM-DD>/`), сохраняя
     относительный путь;
   - `nothing` — ничего не делать.

Область offload (`source_files`) берётся у самого rsync (`rsync --list-only`
источника с теми же `--filter`-правилами: правила профиля + безусловные
артефакты), из листинга — только файлы. Это исключает расхождение движков
сопоставления: нельзя удалить локально файл, который rsync исключил из
передачи. Этот же список — область подтверждения для `after_push`.

Гарантии: никогда не удаляем непереданное (`verify`-подтверждение
пофайловое); ошибки отдельных файлов попадают в `errlist` и поднимают код
профиля до 2, но не прерывают обработку остальных; `verify = false`
подтверждает по факту успешной передачи (без повторной сверки) — допустимо
только при осознанном выборе, дефолт `true`.

Меняя логику offload — синхронизируй `tests/syncher/test_offload.py`, README
и `examples/syncher/`.

Примечание: во внутренней модели допускается маппинг (`archive` →
внутреннее действие `backup`), но внешний TOML-контракт и поведение rsync не
меняются.

## Правила include/exclude и трансляция в rsync

Сопоставление путей выполняет **сам rsync** через `--filter`-правила — он
единственный источник истины и для передачи, и для определения области
offload (через `rsync --list-only`). Отдельного матчера в проекте нет;
`ignore.py` — только детерминированная сборка правил. `pathspec` режимом не
используется.

Синтаксис правил (фильтры rsync, gitignore-подобные): метасимволы `*` (в
пределах сегмента), `**` (cross-segment), `?`, `[abc]`/`[a-z]`, завершающий
`/` (только каталоги), ведущий/срединный `/` (якорь к корню передачи =
`local_root`). `exclude` исключает объект; `include` возвращает его
(override) и выигрывает над `exclude`.

Авто-исключение артефактов (безусловно): `.fs-sync.toml`, `.fs-log`, `.env`
исключаются ВСЕГДА, и `include` их не возвращает — они добавляются первыми в
rsync-фильтры. Иначе они уйдут на сервер, а меняющийся `.fs-log` сломает
идемпотентность.

Трансляция в rsync (`build_filters`/`filter_args`) — rsync обрабатывает
фильтры по принципу «первое совпадение выигрывает», порядок важен:

    1. безусловные артефакты:  - /.fs-sync.toml, - /.fs-log, - .env
    2. include (override):     + <pattern>
    3. exclude:                - <pattern>

Так артефакты не вернуть никаким `include`, а пользовательский `include`
перекрывает `exclude`.

Почему нет своего матчера (важно для безопасности): если бы область offload
(что удалять локально) вычислялась отдельным движком (например, `pathspec`),
его семантика могла бы разойтись с фильтрами rsync на краевых паттернах — и
offload удалил бы локальный файл, который rsync на сервер не отправлял.
Поэтому область offload берётся у самого rsync (`source_files` →
`rsync --list-only` с теми же фильтрами): расхождению движков взяться
неоткуда.

Ограничение: файл нельзя вернуть из каталога, исключённого целиком: rsync в
него не заходит. Возвращайте каталог явно (`include = ["dir/"]`), затем
точечные правила внутри. Паттерны пишутся через `/` без указания диска
(кросс-платформенно).

Любое изменение трансляции фиксируй тестами в `tests/syncher/test_ignore.py`
(и реальный rsync — в `tests/syncher/test_rsync.py`) и отражай в README.
