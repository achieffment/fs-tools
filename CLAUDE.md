## Аудит правок и проекта

Для запросов аудита используется единый контракт — см. skill
`audit-governor` (`.claude/skills/audit-governor/SKILL.md`), запускаемый в
режиме `audit changed` (аудит внесённых правок, дефолт) или `audit full`
(аудит всего репозитория).

Обязательный охват без исключений: код, тесты, examples, документация,
комментарии, правила проекта (`CLAUDE.md`/вложенные `CLAUDE.md`), `AGENTS.md`.

Критерии качества: высокий инженерный уровень, SRP, DRY, минимализм,
простота, читаемость; консистентность между всеми пластами; кроссплатформенность
и безопасность; отсутствие необоснованных suppression-комментариев
(`# pylint: disable`, `# noqa`, `# type: ignore`).

Обязательные точечные проверки (spot-checks):

- dry-run контракт: `normalizer` и `syncher` при `--dry-run` пишут `.fs-log`
  с пометкой `Режим: dry-run` и последовательностью dry-run-событий;
- runner-паттерн: режимы с собственными флагами используют `_build_parser()`;
- одноразовый `path_help` не выносится в отдельную константу без повторного
  использования;
- локальная симметрия имён в критичных блоках: `src_rel`/`dst_rel` в
  `normalizer/engine.py`; `map_norm_argument`/`add_norm_argument`/
  `norm_argv_from_namespace` в `fs_tools/cli.py`.

Обязательный цикл после исправлений — до полного green:

    .venv/bin/python -m pytest -q
    .venv/bin/python -m pylint --persistent=n --recursive=y src tests/*
    .venv/bin/python -m ruff check .
    .venv/bin/python -m mypy --strict -p fs_tools

При расхождении IDE-диагностики и общего `pylint` — точечный запуск:

    .venv/bin/python -m pylint --persistent=n tests/path/to/file.py

Если для прохождения проверок требуется корректировка `CLAUDE.md`, вложенных
`CLAUDE.md` или `AGENTS.md` — изменения вносятся минимально, согласованно и
только по фактической необходимости.

## Консистентность всегда

Код, тесты, примеры и документация — единое целое и не должны расходиться.
Любое изменение поведения синхронизирует все четыре пласта.

Симметрия `normalizer`/`checker` намеренная: ядро каждого — `engine.py` с
классом `Fs*` (`FsNormalizer`/`FsChecker`); локальная переменная инстанса в
`runner.run` — `fsnm`/`fsch`. `syncher` **несимметричен**: нет `engine.py`/
`Fs*`-класса и нет `safety.py` (структура функциональная: `cli_args`/
`config`/`ignore`/`rsync`/`offload`/`report`). Симметрию сохраняют только
врапперы `log.py`/`runner.py` и раскладка тестов/примеров.

Контракт терминального вывода: финальный блок у `normalizer`, `checker` и
`syncher` — две строки: `Статус: <ok|warn|error>. <фраза>` и `Сводка: <метрики
в фиксированном порядке>`. Не возвращать отдельный формат вроде `Готово...`,
если остальные режимы уже используют `Статус:`/`Сводка:`. Изменения формата
синхронизируются между `*/report.py`, `tests/*/test_report.py`,
`tests/*/test_runner.py`, `tests/*/test_examples.py`, `README.md`,
`examples/*/README.md`.

Контракт текста веб-хуков — фиксированные строки:

- checker: `fs-checker - выполнен с ошибкой.`
- syncher: `fs-syncher - выполнен с ошибкой.`

Отправка — fire-and-forget, не влияет на код возврата. checker отправляет
только при непустом `missing`; syncher — только в production-прогоне и только
при наихудшем коде `2`/`3`. Изменения текста синхронизируются между кодом
раннеров, режимными тестами (`tests/checker/test_runner.py`,
`tests/syncher/test_runner.py`) и документацией.

`.fs-log` — намеренное исключение из идемпотентности (дополняется на каждом
запуске), в `.gitignore`; `reset.*` (только у normalizer) убирает его явным
`rm`/`del`. Все режимы пишут журнал в `production` и `dry-run`; в `dry-run`
фиксируется последовательность dry-run-событий (включая конфликты/ошибки), а
откат песочницы syncher — через `git restore`/`git clean` (без `reset.*`).

Локальный шаблон runner-парсеров: если в режиме есть дополнительные CLI-флаги
(как `--dry-run`), парсер собирается в приватной функции `_build_parser()` по
аналогии с `syncher/runner.py`. Для одноразовой подсказки `path_help` в
`_build_parser()` не выноси отдельную константу без повторного использования.

Документация: единый `README.md` (три секции), `examples/README.md` (+
секции режимов). Перед завершением убедись, что ни один пласт не отстал.

Распределение по пакетам (`src/fs_tools/`, дополняет дерево из `AGENTS.md`):
новое правило normalizer регистрируется в `build_normalizer()` и
ре-экспортируется в `rules/__init__.py` и `normalizer/__init__.py`
(`__all__`). Барьер «имя — один компонент пути» (`safety.py`) живёт в
`normalizer/`, а не в `shared/`: checker ничего не переименовывает.

Точки входа и коды возврата:

- normalizer/checker: `runner.main(argv) -> int` + `run(root, ...) -> int`.
  Без аргумента каталог выбирается интерактивно через `pick_directory()`,
  аргумент-каталог минует диалог. У normalizer есть `--dry-run` (план без
  переименований). Коды `0`/`1`/`2` зафиксированы контрактом и проверяются
  тестами.
- syncher: `runner.main(argv) -> int` + `run(root, args) -> int`. Без
  аргумента каталог выбирается интерактивно через `pick_directory()`,
  аргумент-каталог минует диалог. Коды `0/1/2/3` (наихудший среди профилей)
  зафиксированы контрактом.
- Диспетчер `fs_tools.cli:main` (`fs-tools <normalize|check|sync>`) и
  `__main__.py` импортируют `runner` выбранного режима **лениво** (через
  `importlib` в обработчике), чтобы `--help` и доступный режим работали без
  extra другого режима. Подкоманды `normalize` и `sync` объявляют флаги
  режима и пробрасывают их (а не только `path`).

## Кросс-платформенная безопасность

Код безопасно исполняется на Windows, WSL, Linux и macOS. Файловые операции
не должны приводить к потере файлов и данных.

- **Не выходить за рамки**: normalizer переименовывает только содержимое
  выбранного каталога; сам корень не трогаем, по скрытым (`.`) объектам не
  заходим. checker структуру не меняет вовсе (единственная запись —
  `.fs-log`).
- **Имя — один компонент пути**: разделители (`/`, `\`), управляющие и
  запрещённые на Windows символы (`< > : " | ? *`) не должны попадать в
  `os.rename`. Чистит `TransliterationRule`, страховочный барьер —
  `normalizer/safety.py` (`enforce_safe_component`), вызывается в `name.py`.
  Не ослабляй их.
- **Без перезаписи**: занятый другим объектом `dest` — конфликт, элемент
  пропускаем; конфликт учитывается в `FsNormalizer.conflicts` (безопасный
  пропуск) и на код возврата не влияет.
- **Порядок**: переименование deepest-first (дети раньше родителей).
- **Case-only** переименование на регистронезависимых ФС — через временное
  имя.
- **Журнал `.fs-log`** (`shared/log.py`, `append_log`) пишется в корень
  выбранного каталога только режимом append (`open("a")`, `utf-8`) и под
  скрытым именем — он ничего не перезаписывает и сам обходом пропускается.
- **Выбор каталога** (`shared/picker.py`): нативный диалог Windows/WSL
  (`IFileOpenDialog` через `powershell.exe` + `pick_folder.ps1`, `wslpath`,
  проброс через `WSLENV`), `osascript` на macOS, ввод в терминале на обычном
  Linux; при сбое любого диалога — откат в терминал. Сохраняй эти fallback'и.
- **Ресурс `pick_folder.ps1`** ищется через
  `importlib.resources.files("fs_tools.shared")`, НЕ вычислением пути от
  `__file__` (иначе не найдётся в установленном wheel). То же для `.env`
  (`shared/env.py`): путь через `FS_TOOLS_HOME`/CWD, не от `__file__`;
  загрузка — `load_dotenv(override=False)` (переменные процесса важнее
  значений файла).
- **Синхронизация**: однонаправленность ПК → сервер; без offload-профиля
  локальные файлы не удаляются. Пути для rsync — posix (`as_posix()`);
  источник всегда с завершающим `/`. Windows: нативного `rsync` нет — работа
  через WSL или cwrsync. Артефакты (`.fs-sync.toml`, `.fs-log`, `.env`) не
  передаются никогда. Веб-хук `FSSYN_*` — только `https://`.

Осознанные допущения (не баги):

- Зарезервированные имена Windows (`CON`, `NUL`, `COM1..9`, `LPT1..9`) и
  превышение длины пути специально не обезвреживаются: `os.rename` падает,
  ошибка ловится, элемент пропускается (данные целы), фиксируется в
  `FsNormalizer.errlist`, код возврата `2`.
- Case-only идёт в два шага через временное имя без отката: объект остаётся
  под скрытым `.__normtmp_*` при сбое второго шага.
- На POSIX `os.rename` молча перезатирает `dest`; защита — предварительная
  проверка занятости. Конкурентный доступ к ФС вне модели угроз.
- Сбой записи `.fs-log` — вторичный артефакт: только предупреждение в
  `stderr`, на код возврата не влияет.

## Комментарии — кратко и по существу

Объясняй неочевидное (зачем, ограничения, компромиссы), не пересказывай код.

**Код (Python, PEP 8)**: inline-комментарий — минимум 2 пробела до `#`, затем
`# ` и текст. Докстринги и комментарии — на русском, лаконично.

**Документация (все `*.md` проекта)**: inline-комментарии в **командных**
fenced-код-блоках (`bash`, `sh`, `shell`, `powershell`, `pwsh`, `bat`, `cmd`)
держим на одном уровне (выровнены в одну колонку):

- выравнивание считается по локальному подблоку между пустыми строками;
- опорная длина берётся по самой длинной непустой строке подблока (кроме
  строк, начинающихся с `#` после отступов);
- в строке с inline-комментарием отступ до `#` — ровно 4 пробела от этой
  опорной длины; в остальных строках добавляй пробелы так, чтобы `#` стоял в
  той же колонке;
- опорная колонка = `длина_самого_длинного_текста + 4`, не выравнивай по
  случайно завышенному отступу.

Правило действует для всех markdown-документов репозитория (`README.md`,
`AGENTS.md`, `examples/*`, `docs/*` и т. д.) и автоматизировано тестом
`tests/shared/test_markdown_comments.py` — обязательная часть `pytest`-цикла.

    pytest                          # тесты
    ruff check .                    # линтер
    mypy --strict -p fs_tools       # типы

## Импорты — по PEP 8 (isort), а не по использованию

Порядок импортов идиоматичный (PEP 8/isort), не «по порядку использования».
Блок импортов — справочник, а не сценарий исполнения.

- Группы (между группами — пустая строка), в порядке: `from __future__ import
  ...` → стандартная библиотека → сторонние пакеты → локальные/относительные
  модули проекта (`from . import ...`, `from ..shared ...`).
- Внутри группы — по алфавиту; `import x` идёт перед `from x import y`.
- Исключение — ре-экспортирующие `__init__.py`
  (`fs_tools/normalizer/__init__.py`, `fs_tools/checker/__init__.py`,
  `rules/__init__.py`): порядок `__all__` отражает логику API/конвейера.
  Сами import-строки в них всё равно сортируются isort.
- **Ленивые импорты — намеренные**: тяжёлые/опциональные зависимости
  (`unidecode`, `requests`, `python-dotenv`) и runner режимов в диспетчере
  импортируются внутри функций, а не на уровне модуля. Не поднимай их в
  шапку файла.

Конвенция включена в `ruff` правилом `I` (isort) в `ruff.toml`:

    ruff check .                    # проверка (в т.ч. порядок импортов)
    ruff check --fix .              # автоисправление порядка

Не переставляй импорты вручную «по смыслу» — доверься `ruff --fix`.

## Порядок lazy-import (`importlib.import_module`)

В блоке последовательных загрузок через `importlib.import_module` порядок
строк должен совпадать с порядком первого использования загруженных имён
ниже по коду.

- Область применения: только конструкции вида `name =
  importlib.import_module(...).attr`, только последовательные блоки
  присваиваний внутри одной функции.
- Критерий: если имя `a` впервые используется раньше `b`, строка загрузки `a`
  должна идти раньше строки загрузки `b`. Одиночная загрузка не проверяется.
- Исключения: независимые ветви `if/elif/else` (порядок использования
  зависит от ветки); имя, ни разу не используемое в текущем теле функции.
- Цель: сверху видно те зависимости, которые сразу идут в работу в том же
  порядке — упрощает чтение `run`-функций.

## Именование и симметрия переменных

Применяй единый словарь и симметрию длин для логически связанных локальных
имён. Цель — визуальная ровность соседних переменных без потери смысла.

Базовый словарь замен:

- `local -> source`, `remote -> target`, `archive -> backup`, `archive_dir ->
  backup_dir` (внешний TOML → внутренние поля), `raw -> bare`, `dir -> path`
  (по смыслу), `local_root -> source_root`, `remote_root -> target_root`
  (внешний TOML → внутренняя модель), `stamp -> date`, `content -> cont`,
  `token -> tok`, `separator -> sep`, `slash -> slh`, `profiles -> roll`,
  `process -> proc`, `title -> header`, `convert -> conv`, `returncode -> rc`
  (кроме stdlib-контрактов), `current -> curr`, `reports -> result`,
  `operations -> actions`, `confirmed -> confirm`, `offloaded -> offload`,
  `errors -> errlist`, `drive -> disk`.

Допустимые сокращения из истории проекта:

- `target -> targ` (например `resolve_root(targ)`);
- пары `source/target -> src/dst` в плотных локальных блоках;
- `source_path/target_path -> srcp/dstp` при длинных цепочках путей;
- `index/left/right/middle/right_tail -> ix/lf/rt/md/rtail`;
- `anchor_count -> anccnt`, `include -> incl`, `regex -> rx`, `log_path ->
  lpath`, `shared/managed/converted -> shar/mang/con|conv`;
- инстансы режимов: `fsnm` (normalizer), `fsch` (checker) — те же имена
  применяются и в тестах, не только в `runner.py`.

Локальные пары для CLI-диспетчера (`fs_tools/cli.py`) — частный случай общего
правила локальной симметрии:

- normalizer-блок: `map_norm_argument` / `add_norm_argument` /
  `norm_argv_from_namespace`;
- sync-блок: `map_sych_argument` / `add_sync_argument` /
  `sync_argv_from_namespace`;
- не смешивай в одном блоке формы `normalizer_*` и `norm_*` без причины.

Локальные пары путей в `normalizer/engine.py`: относительные пути в одном
блоке — симметричная пара `src_rel` / `dst_rel` (не `src_rel` / `dest_rel`).

Правило подбора имени:

- выравнивай только логических «соседей» в одном блоке/функции, не выравнивай
  «вообще всё» — локальная симметрия важнее глобального переименования;
- однотипные переменные с нумерацией (`name1`, `name2`) — нумерация
  обязательна для всей пары/серии, не смешивай `name` и `name2`;
- предпочтение: (1) полная читабельная форма (`source/target`,
  `confirm/offload/errlist`); (2) краткая симметричная форма (`src/dst`,
  `sep/slh`, `url/tok`), если модуль уже в таком стиле;
- в пределах одного модуля не смешивай хаотично полные и краткие формы одной
  сущности.

Исключения (не переименовывать): контракты stdlib и внешних API
(`proc.returncode`, `CompletedProcess.returncode`); публичные контракты,
требующие отдельной миграции (внешние CLI/env/TOML поля), если задача явно не
про миграцию; устойчивые имена протоколов/колбэков (`os.walk`:
`dirpath/dirnames/filenames`, `Path.is_dir`); текстовые данные
фикстур/примеров, где слова являются содержимым, а не именами переменных.

Примеры: хорошо — `source_path`/`target_path`; `sep`/`slh`; `result`/
`actions`; `confirm`/`offload`/`errlist` (тройка одинаковой длины). Плохо —
смешивать `target` и `targ` в одном небольшом блоке без причины.

## Тесты режимов

Дополняет раздел «Тесты» в `AGENTS.md`. Режим запуска —
`--import-mode=importlib` (задан в `pyproject.toml`); одноимённые
`test_engine.py` / `test_ignore.py` / `test_log.py` / `test_runner.py` /
`test_notify.py` сосуществуют между режимами (importlib). Хак вставки корня
в `sys.path` не нужен (src-layout + editable).

Каталоги тестов обязаны быть пакетами Python: в `tests/`, `tests/shared/`,
`tests/normalizer/`, `tests/normalizer/rules/`, `tests/checker/`,
`tests/syncher/` должен быть `__init__.py`. Новые тестовые подпапки создавай
сразу с `__init__.py`. Без этого `pylint` может обходить дерево неполно даже
при `--recursive=y`.

- Корневой `tests/conftest.py` — только по-настоящему общие фикстуры: дерево
  `make_tree` (фабрика из списка путей в `tmp_path`). Мод-специфичные
  держатся в `conftest.py` подпапок: фабрика нормализатора `nn`
  (`build_normalizer`) — `tests/normalizer/conftest.py`; запись `.fs-check`
  `write_rule` — `tests/checker/conftest.py`; `make_tree(base, paths)`
  (переопределяет общий под деревья источника/приёмника) + `write_config` —
  `tests/syncher/conftest.py`. Маркер `requires_rsync` (skip без бинаря
  rsync) определяется локально в нуждающихся тест-файлах `syncher`.
- `tests/shared/test_picker.py` — один общий файл: выбор каталога не
  дублируется.
- `tests/shared/test_env.py` — доступ к единому `.env`: путь, загрузка в
  `os.environ` (`load_env`), приоритет «процесс > файл» (`override=False`),
  идемпотентность, упрочнение прав. `load_env` мутирует `os.environ`,
  поэтому autouse-фикстура сбрасывает `env._STATE["loaded"]` и
  восстанавливает окружение. Веб-хук-ключи проверяются у checker.
- `tests/shared/test_log.py` — общие механики журнала (метка времени,
  отступ, append, utf-8). Мод-специфичные строки журнала проверяются в
  мод-тестах: пары `old -> new` и «(изменений нет)» — у normalizer; пути и
  «(нарушений нет)» — у checker; маркеры `+`/`-`/`>>` и «(изменений нет)» —
  у syncher.
- `tests/shared/test_notify.py` — общая логика веб-хука (`shared.notify`):
  чтение конфигурации из env, https-only, lazy `requests`, Bearer-заголовок,
  гашение сетевых ошибок.

Обязательные виды проверки:

- **normalizer**: юнит-правила (`tests/normalizer/rules/test_<rule>.py`),
  пайплайн `name.py`, идемпотентность (`normalize(normalize(x)) ==
  normalize(x)`), безопасность имени (`enforce_safe_component`), фильтр
  `.fs-ignore`, `FsNormalizer`, коды возврата `runner.main`.
- **checker**: парсинг `.fs-check`, разворачивание правил, коды возврата,
  picker, веб-хук (мок сети).
- **syncher**: парсинг/валидация `.fs-sync.toml` и коды `ConfigError`;
  трансляция include/exclude в фильтры rsync; разбор
  `--itemize-changes`/`--list-only`; delete-guard (пороги по
  количеству/доле, код 3); offload (verify → `after_push`, частичный
  успех); коды возврата `0/1/2/3` и «наихудший среди профилей»; «нет
  аргумента → интерактивный picker, аргумент-каталог минует диалог».
  Интеграционные тесты с реальным rsync — `@requires_rsync` (skip без
  бинаря).
- **Аргумент-каталог минует диалог** — для всех режимов; при отсутствии
  аргумента normalizer/checker/syncher используют интерактивный выбор
  каталога.
- **Веб-хук**: окружение процесса важнее `.env` (проверяется в
  `test_env.py`); checker/syncher-тесты проверяют ключи окружения и
  делегирование в `shared.notify`, а поведение отправки (https-only, lazy
  `requests`, заголовки, гашение ошибок) проверяется в
  `tests/shared/test_notify.py`.

`pylint` запускаем без кэша (`--persistent=n`) и с рекурсивным discovery
(`--recursive=y`) всегда. Общий прогон выполняем по шаблону `src tests/*`,
чтобы проверялись все режимы тестов; все тестовые каталоги при этом должны
быть пакетами с `__init__.py`. При любом расхождении между диагностикой IDE
и общим прогоном обязателен точечный запуск по проблемному файлу:

    .venv/bin/python -m pylint --persistent=n tests/path/to/file.py

Только после общего + точечного прогона разрешено фиксировать статус
«ошибок нет».

Всё должно быть зелёным. Идемпотентность дополнительно проверяй на копии
`examples/normalizer/` (исходник под git не трогаем):

    cp -r examples/normalizer /tmp/ex && rm -f /tmp/ex/.fs-ignore
    .venv/bin/python -c "from pathlib import Path; from fs_tools.normalizer import \
    build_normalizer, FsNormalizer as F; r=Path('/tmp/ex'); \
    n=F(build_normalizer()); n.apply(r); print(n.apply(r))"

Второй прогон **ничего не переименовывает** — ожидаем `(0, 1)`: единственный
`skipped` — задокументированный конфликт `08-edge-cases/conflict`. Любое
`renamed > 0` на втором прогоне — регресс идемпотентности.

Демо-инвариант checker: прогон на `examples/checker/` даёт ровно ожидаемое
число нарушений (см. `examples/checker/README.md`).

Демо-инвариант syncher: `--dry-run` на `examples/syncher/` даёт
зафиксированный итог (см. `examples/syncher/README.md`; проверяется
`test_examples.py`, skip без rsync).
