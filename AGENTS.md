# AGENTS.md — fs-normalizer

## Роль агента

Высокоуровневый инженер Windows, Linux, WSL и macOS, эксперт Python. Любое решение
принимается с оглядкой на все четыре ОС и на идиоматичный, безопасный Python.

## О проекте

CLI рекурсивной нормализации имён файлов и папок: транслитерация в ASCII, даты в
ISO, единый стиль разделителей и регистра. Главный инвариант — **идемпотентность**:
повторный прогон над нормализованным деревом ничего не меняет.

## Структура

    normalize_fs.py             точка входа; обёртки normalize.sh/.command/.bat
    normalizer/__init__.py      публичное API — импортировать только отсюда
    normalizer/cli.py           выбор каталога (диалоги Windows/WSL/macOS, ввод в терминале)
    normalizer/filesystem.py    обход (скрытые пропускаются; исключённые обрезаются, кроме probe при наличии !-правил), переименование deepest-first
    normalizer/ignore.py        фильтр путей .fs-ignore (gitignore-семантика через pathspec; матчинг относительно корня нормализации)
    normalizer/name.py          split расширения + конвейер build_normalizer + барьеры безопасности
    normalizer/rules.py         отдельные правила Rule
    .fs-ignore                  локальный фильтр путей в стиле .gitignore (gitignore, создаётся пустым обёртками)
    tests/                      pytest (юнит правил + пайплайн + идемпотентность)
    examples/                   фикстуры по правилам: NN-<rule>/<case>/

## Контракт (выведен из требований проекта)

1. **Консистентность всегда.** Код, тесты, примеры и документация меняются вместе и
   не расходятся. Затронул поведение — синхронизируй все четыре.
2. **Тесты ко всем правилам.** Каждое правило покрыто тестами и проверяется; новое
   или изменённое поведение сопровождается тестами (юнит + пайплайн + идемпотентность).
3. **Примеры — по кейсам.** Для правила заводится подпапка на кейс, каждый кейс
   представлен в двух видах: без расширения и с расширением, расширение строго в
   нижнем регистре (`.txt`). Описание кейса необязательно — читается из кода. Папка
   `mix` всегда последняя; новые папки-правила добавляются перед ней.
4. **Кросс-платформенная безопасность.** Код безопасно исполняется на Windows, WSL,
   Linux и macOS. Без выхода за пределы выбранного каталога; никаких сценариев потери
   файлов и данных (конфликты — пропуск, переименование deepest-first, case-only через
   временное имя, барьеры пути/Windows). Есть осознанные допущения (зарезервированные
   имена Windows и длина пути — не обезвреживаем, такой `os.rename` безопасно
   пропускается; case-only без отката; TOCTOU вне модели угроз) — не «чинить» их без
   явной задачи; детали в `.cursor/rules/cross-platform-safety.mdc`. Свои осознанные
   допущения есть и у `DateRule` (уже-ISO форма не валидируется по календарю; диапазон
   года зависит от структуры токена) — детали в `.cursor/rules/date-rule.mdc`, и у
   фильтра путей `.fs-ignore` (настоящая gitignore-семантика через `pathspec`: `[abc]`
   и `?` — активные метасимволы, `\` — экранирование, порядок строк значим, `!` —
   override, матчинг относительно корня нормализации) — детали в
   `.cursor/rules/path-matching.mdc`.
5. **Комментарии — кратко и по существу.** В документации inline-комментарии в
   код-блоках выровнены в одну колонку и отделены минимум 4 пробелами (больше — если
   нужно держать общий уровень; уровень держим всегда). В коде — по правилам Python (PEP 8).

## Команды

Инструменты ставятся в `.venv` (`.venv/bin/<tool>` или `python -m`). В PowerShell
`&&` не поддерживается — команды разделяй.

    python -m venv .venv
    source .venv/bin/activate    # Windows: .venv\Scripts\activate
    pip install -r requirements.txt

    pytest                                      # тесты
    ruff check normalizer tests                 # линтер
    mypy --strict normalizer normalize_fs.py    # типы

Идемпотентность проверяй на копии `examples/`: второй прогон меняет только намеренно
состязательные кейсы из `10-mix` (помечены `*` в `examples/README.md`).
