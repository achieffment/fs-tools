# AGENTS.md — fs-checker

## Роль агента

Высокоуровневый инженер Windows, Linux, WSL и macOS, эксперт Python. Любое решение
принимается с оглядкой на все четыре ОС и на идиоматичный, безопасный Python.

## О проекте

CLI-проверка структуры каталогов по правилам из `.fs-rule` (gitignore-подобный
синтаксис с инвертированным смыслом: правила задают **обязательные** пути). Утилита
берёт корень проверки (интерактивный диалог или аргумент-каталог для cron), читает
из него `.fs-rule`, рекурсивно сверяет наличие папок и файлов и печатает
отсортированный список отсутствующих путей + сводку. Главный инвариант —
**структура дерева не меняется**: утилита не создаёт, не переименовывает и не
удаляет файлы/папки (создание недостающего — вне области задачи). Единственная
запись на диск — журнал `.fs-log` в проверяемом каталоге при наличии нарушений
(зеркало `fs-normalizer`); при нарушениях возможен и сетевой побочный эффект —
fire-and-forget веб-хук-уведомление (`.env`).

## Структура

    check_fs.py                   точка входа; обёртки check.sh/.command/.bat (авто-venv + pip)
    checker/__init__.py           публичное API — импортировать только отсюда
    checker/cli.py                разбор аргументов (опц. аргумент-каталог), выбор каталога, загрузка правил, печать, запись .fs-log и веб-хук при нарушениях, коды возврата
    checker/picker.py             выбор каталога (диалоги Windows/WSL/macOS, ввод в терминале, wslpath-конверсии)
    checker/pick_folder.ps1       нативный диалог Windows (IFileOpenDialog)
    checker/rule.py               разбор .fs-rule: положительные правила (glob-строки) + PathSpec негативов; FsRuleError при отсутствии файла
    checker/pathspec_compat.py    совместимость версий pathspec: выбор фабрики gitignore-паттернов (_FACTORY)
    checker/engine.py             FsChecker.check(root): разворачивание префиксов через Path.glob, прунинг негативов по *-/**-выбранным компонентам, проверка мандата (exists()/is_dir()), сбор отсутствующих путей
    checker/report.py             формат отчёта и сводки
    checker/log.py                журнал .fs-log (метка времени + отсутствующие пути); формат зеркалит fs-normalizer
    checker/notify.py             веб-хук о нарушениях: load_webhook_config из .env (FSCHK_WEBHOOK_*), fire-and-forget send_webhook (requests, мин. таймаут)
    tests/                        pytest: test_rule/test_engine/test_cli/test_picker/test_log/test_notify + conftest
    examples/                     запускаемая песочница-фикстура + .fs-rule + README

## Контракт

1. **Структуру не меняем.** Утилита не создаёт/не переименовывает/не удаляет
   папки и файлы проверяемого дерева. Разрешённые записи — только журнал `.fs-log`
   в проверяемом каталоге при нарушениях (формат как у `fs-normalizer`) и вывод в
   stdout/stderr. Веб-хук — сетевой побочный эффект, тоже только при нарушениях.
2. **Консистентность всегда.** Код, тесты, `examples/` и документация меняются
   вместе и не расходятся. Затронул формат `.fs-rule` или семантику проверки —
   синхронизируй все четыре пласта (см. `.cursor/rules/consistency.mdc`).
3. **Семантика правил.** Префикс = существующие якори (`Path.glob`), последний
   сегмент = мандат (обязателен). Завершающий `/` → `is_dir()`, иначе `exists()`.
   Негатив `!` прунит только подстановки `*`/`**` по имени **каждой** выбранной
   компоненты (а не только листа) и не влияет на литералы; узкий момент `_Archive`
   и фиксированные каталоги — в `.cursor/rules/rule-matching.mdc`. Пустой
   префикс → якорь = `root` (не `glob(".")`).
4. **Кросс-платформенная безопасность.** Picker с fallback в терминал на всех ОС;
   не выходим за пределы выбранного каталога; кодировки/EOL/исполняемые биты — в
   `.cursor/rules/cross-platform-safety.mdc`.
5. **Тесты ко всему.** Парсинг, разворачивание, коды возврата CLI, picker, журнал
   `.fs-log` и веб-хук покрыты тестами (`.cursor/rules/testing.mdc`).
6. **Импорты — по PEP 8 (isort)**, не по использованию; правится `ruff check --fix`
   (`.cursor/rules/imports.mdc`). Комментарии — кратко, объясняют неочевидное
   (`.cursor/rules/comments-style.mdc`).

## Команды

Инструменты ставятся в `.venv` (`.venv/bin/<tool>` или `python -m`). В PowerShell
`&&` не поддерживается — команды разделяй.

    python -m venv .venv
    source .venv/bin/activate    # Windows: .venv\Scripts\activate
    pip install -r requirements.txt

    pytest                                  # тесты
    ruff check checker tests                # линтер
    mypy --strict checker check_fs.py       # типы

Прогон на песочнице: запусти утилиту на `examples/` — отчёт обязан совпасть с
зафиксированным в `examples/README.md` (ровно 7 отсутствующих путей).
