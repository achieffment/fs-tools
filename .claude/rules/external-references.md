# Самодостаточность: без ссылок на внешние проекты-источники

> Claude-эквивалент [`.cursor/rules/external-references.mdc`](../../.cursor/rules/external-references.mdc). Применяется при работе с `**/*.py`, `**/*.md`, `**/*.mdc`, `examples/**`.

Проект собирался из нескольких самостоятельных репозиториев (в т.ч. проектов,
которые были источниками переиспускания кода/правил/README-формата на этапе
слияния и последующей разработки по ТЗ). После переноса в `fs-tools` эти
источники — **внешние** проекты: они не видны читателю репозитория, не
публикуются вместе с ним и могут измениться или исчезнуть независимо. Любая
живая ссылка на них делает `fs-tools` несамодостаточным.

## Запрет

Во всех пластах проекта — коде, тестах, примерах, документации (`README.md`,
`AGENTS.md`), правилах (`.cursor/rules/*.mdc`/`.claude/rules/*.md`), скиллах
(`.cursor/skills/`/`.claude/skills/`) — запрещены:

- упоминания внешних проектов-источников переиспользования по имени как
  живого референса («сделано как в `<внешний-проект>`», «см. `<внешний-проект>`
  README», сравнение форматов между `fs-tools` и внешним проектом и т.п.);
- гиперссылки/URL на такие внешние репозитории;
- локальные файловые пути, указывающие на каталоги этих проектов на диске
  разработчика (например `/home/.../Components/<внешний-проект>`).

Если что-то взято из внешнего проекта-источника (правило, кусок README,
паттерн кода) — оно **адаптируется и переносится внутрь** `fs-tools` целиком,
без указателя на происхождение. Проект должен читаться и пониматься сам по
себе, без обращения к внешним каталогам/репозиториям.

## Не считается нарушением

- Ссылки/имена **этого же** дистрибутива — `fs-normalizer`, `fs-checker`,
  `fs-syncher`, `fs-schemer`, `fs-tools` (console_script-команды, имена
  режимов, тексты веб-хуков, `prog=` в `argparse`) не являются внешними
  проектами: это собственная публичная идентичность `fs-tools`, а не
  переиспользованный внешний источник.
- Ссылки на реальные сторонние библиотеки/зависимости, используемые как
  обычный пакет (`pathspec`, `rsync`, `python-dotenv`, `unidecode`,
  `requests` и т.п.) — это внешние зависимости дистрибутива, а не проекты,
  из которых код/правила переносились вручную.
- Каталог `docs/` — не отслеживается git (см. `.gitignore`), содержит только
  внутренние рабочие ТЗ/промпты разработки и не публикуется вместе с
  репозиторием; это правило на него не распространяется. Уже попавшие туда
  упоминания/пути менять не требуется.

## Проверка перед завершением задачи, где мог появиться внешний референс

- [ ] Нет строк с именами внешних проектов-источников за пределами `docs/`.
- [ ] Нет локальных путей вида `/home/.../Components/<внешний-проект>` за
      пределами `docs/`.
- [ ] Заимствованный фрагмент (правило/README/код) переписан как
      самостоятельный текст `fs-tools`, без «см. внешний проект».
