# Семантика `.fs-chk` и проверки

> Claude-эквивалент [`.cursor/rules/rule-matching.mdc`](../../.cursor/rules/rule-matching.mdc). Применяется при работе с `src/fs_tools/checker/rule.py`, `src/fs_tools/checker/engine.py`.

`.fs-chk` — gitignore-подобный синтаксис с **инвертированным** смыслом: строки
задают пути, которые ОБЯЗАНЫ существовать. Разбор — `fs_tools/checker/rule.py`,
проверка — `fs_tools/checker/engine.py`.

## Парсинг (rule.py)

- Кодировка `utf-8-sig`. Комментарий — только **ведущий** `#` (инлайн `#` —
  литерал). Пустые строки игнорируются. Конечные пробелы обрезаются, кроме
  экранированных `\ `. Разделитель сегментов — POSIX `/`.
- Пропуск комментариев/пустых строк и классификацию (положительные / `!`-негативы)
  делает сам `rule.py`; в `pathspec` уходят только шаблоны без ведущего `!`.
- **Завершающий `/`** распознаётся ДО снятия якорного `/` и делает мандат строго
  каталогом (`is_dir()`); без него — `exists()` (файл ИЛИ папка).
- Нет `.fs-chk` → `FsRuleError` (некорректный запуск, код возврата 1).

## Разворачивание (engine.py)

- Правило = **префикс** (все сегменты, кроме последнего — множество существующих
  якорей) + **последний сегмент** (мандат, обязателен). Префикс без `**` (или с
  несколькими `**`) разворачивается `pathlib.Path.glob`; префикс с ровно одним
  `**` — собственным scandir-обходом (`_expand_anchors_recursive`, быстрее на
  больших деревьях, не разыменовывает symlink). Итоговый набор якорей одинаков у
  обеих стратегий: скрытые каталоги отбрасываются после разворачивания.
- Нарушение — отсутствие мандата в найденном якоре. Если префикс не дал якорей — **не**
  нарушение.
- **Пустой префикс** (правило из одного сегмента, напр. `/Activities`) → единственный
  якорь = сам `root`. НЕ вызывать `root.glob(".")` (ValueError на Python 3.13+).
- **Фиксированные промежуточные каталоги** должны иметь собственные литеральные
  правила — иначе их отсутствие маскирует более глубокие нарушения.
- Мандат сверяется **литерально** (без glob в последнем сегменте). Скрытые каталоги
  (на `.`) не обходим; симлинки не разыменовываем; нарушения дедуплицируются и
  сортируются.

## Негативы `!` (ordered pathspec)

- `!`-шаблоны компилируются в `pathspec.PathSpec` (фабрика `_FACTORY` из
  `fs_tools/shared/pathspec_compat.py`).
- Все строки `!` в `.fs-chk` (после снятия первого `!`) уходят в **единый ordered**
  pathspec-список; спец-веток по типу паттерна нет.
- Матч выполняется по относительным путям якоря и целевого мандата:
  `anchor` и `anchor/require`.
- Поддерживаются обычные pathspec/gitignore-маски (`**`, `*`, `?`, `[]`) и порядок
  строк (`last-match-wins` для конфликтующих паттернов).
- Короткая форма `!_Archive` валидна как pathspec-шаблон basename и отсекает ветки
  `_Archive` на любой глубине; явный эквивалент — `!**/_Archive/**`.
- В checker `!` трактуется только как исключение: re-include не используется.
  Ведущие `!` схлопываются (`!!/a` == `!/a`).
