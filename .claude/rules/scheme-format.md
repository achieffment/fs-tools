# Формат `.fs-sch.toml` и модель движка `schemer`

> Claude-эквивалент [`.cursor/rules/scheme-format.mdc`](../../.cursor/rules/scheme-format.mdc). Применяется при работе с `src/fs_tools/schemer/*`.

Конфиг `.fs-sch.toml` лежит в каталоге, который передаётся режиму (аргумент
`run(root)` либо `pick_directory()`) — по аналогии с `.fs-syn.toml`/`.fs-chk`.
По умолчанию этот же каталог и обходится проверкой. Разбор — `tomllib`, модель —
`schemer/config.py`.

## Раздельный конфиг и проверяемое дерево (`apply_root`)

- `[defaults].apply_root` (строка, опционально) — путь каталога, который реально
  обходится и проверяется, если он отличается от каталога с `.fs-sch.toml`.
  Решает кейс, когда конфиги/логи всех режимов удобно держать в одном месте
  (`E:/Home`), а проверяемое дерево лежит глубже (`E:/Home/Workspace/Warehouse`).
- Разрешение пути — забота `runner.py`, не `config.py`: `parse_scheme_config`
  хранит `apply_root` как сырую строку (без обращения к диску), чтобы оставаться
  тестируемой без файловой системы. `runner.run()` резолвит её: абсолютный путь —
  как есть, относительный — от каталога конфига (`root`); каталог должен
  существовать (`resolve_root`, симметрично проверке самого `root`), иначе код
  возврата `1`.
- Не задано (дефолт) → каталог проверки = каталог конфига (текущее поведение,
  полностью обратно совместимо).
- `FsSchemer.check` обходит и `format_report` печатает (`Каталог:`) именно
  разрешённый `apply_root`, а `.fs-log.log` при этом пишется рядом с конфигом
  (`root`), а не в `apply_root` — это осознанная асимметрия с
  `normalizer`/`checker`: она позволяет держать журналы всех режимов в одном
  общем каталоге (мотив всей фичи), а не рассеивать их по проверяемым деревьям.

## Модель предметной области

Дерево состоит из вложенных тематических узлов произвольной глубины. Любой
тематический узел может содержать ноль или более групповых папок — подпапок с
именем из фиксированного (в конфиге) списка. Групповая папка матчится **по
basename на любой глубине** (как якорь в checker/normalizer), регистр
сравнивается **литерально** (в отличие от `.fs-nrm` у normalizer). Внутри
групповой папки: обязательный служебный файл, опциональные служебные файлы
(проверяются контентно только при наличии) и обычные файлы (не начинающиеся с
`exclude_prefix`, дефолт `_`).

## Секции `.fs-sch.toml`

- `[defaults].exclude_prefix` — префикс служебных файлов группы (дефолт `_`),
  задаётся один раз.
- `[[group]].name` — basename групповой папки; непустой, без `/`, уникален среди
  групп.
- `[[group]].default_rule` — необязательная таблица `{line, text}`: контент-правило
  для «обычных» файлов группы (файлов без записи `group.file`, не начинающихся с
  `exclude_prefix`). Не задано → обычные файлы группы не проверяются контентно.
  - `extensions` (список строк, опционально) — whitelist: под контент-проверку
    попадают только файлы с этими расширениями (напр. `[".md"]`); остальные не
    читаются вовсе и не входят в `files_checked` (не нарушение).
  - `exclude_extensions` (список строк, опционально) — blacklist: под
    контент-проверку попадают все файлы, КРОМЕ перечисленных расширений.
  - Независимы и комбинируются через «И» (можно задать оба сразу): `extensions`
    задаёт стартовый набор («только эти»), `exclude_extensions` из него дополнительно
    убирает перечисленные — итог «whitelist минус blacklist». Каждое поле само по
    себе опционально: не задано `extensions` → старт со «всех файлов группы»; не
    задано `exclude_extensions` → из набора ничего не убирается; не заданы оба →
    читаются все обычные файлы группы (прежнее поведение). Каждый элемент — строка
    с ведущей точкой (`.md`, не `md`); сравнение с реальным расширением файла
    регистронезависимо. Мотив: без фильтра `default_rule` пытается прочитать как
    текст любой файл группы (включая бинарные — `.docx`, `.pdf`, `.bin`), что даёт
    ложный `read_error`; фильтр позволяет ограничить контент-проверку только
    текстовыми форматами.
- `[[group.file]]` — единственный механизм для конкретного `name`: обязательность
  и контент-правило одной записью.
  - `optional` (bool, дефолт `false`): `false` — отсутствие файла →
    `missing_group_file`; `true` — отсутствие не нарушение, наличие требует
    контент-проверки.
  - `line`/`text` обязательны в каждой записи (запись без контент-правила не имеет
    смысла — выразить «просто существование» можно только исключив запись).
  - `name` уникален в пределах группы; отдельного списка `required` нет — это
    единственный механизм обязательности.
- `line` — 1-based номер строки; `text` — точное посимвольное совпадение (без
  конечного перевода строки). Короче `line` строк → `missing_line`.
- `[[group]].strict` (bool, дефолт `false`) — `false` (по умолчанию): группа —
  непрозрачный контейнер, вложенность внутри неё разрешена (сторонние библиотеки,
  произвольная организация); `group.file`/`default_rule` по-прежнему проверяются
  для файлов НЕПОСРЕДСТВЕННО в папке группы, но обход не спускается в её подпапки —
  F15 (`loose_file`) на них не срабатывает. `true` — прежнее строгое поведение:
  подпапки группы заново классифицируются наравне с остальным деревом, F15 в них
  работает как обычно.

## Модель движка (read-only, `schemer/engine.py`, класс `FsSchemer`)

- Обход — `os.walk` от корня без `followlinks` (симлинки не разыменовываются);
  скрытые (на `.`) каталоги и файлы пропускаются, как у normalizer/checker.
- Классификация узла: каталог — «групповой», если его basename совпадает с
  `[[group]].name`; иначе — «тематический».
- **Loose file (файл вне групповой папки)**: для каждого тематического узла — видимые
  файлы напрямую в нём → нарушение на каждый такой файл. Сам `.fs-sch.toml` в
  корне проверки под неё не попадает — он скрытый (ведущая точка) и уже отсеян
  общим фильтром скрытых объектов, без отдельного исключения по имени.
- **Пустая группа**: групповая папка без единого видимого файла (в т.ч. во вложенных
  подпапках, рекурсивно) → нарушение. Это единственная проверка с рекурсивным
  охватом вложенных подпапок — `group.file`/`default_rule` применяются только к
  файлам НЕПОСРЕДСТВЕННО внутри групповой папки.
- **`group.file`**: без `optional` — отсутствие файла → нарушение; `optional=true` —
  при наличии контент-проверка обязательна.
- **Контент-проверка**: `utf-8-sig`, `splitlines()`, `lines[line - 1] == text`
  (индекс отсутствует → `missing_line`, текст не совпал → `bad_header`). Файл не
  удалось прочитать (`OSError`/`UnicodeDecodeError` — нет прав, гонка удаления,
  не-UTF-8 содержимое) → отдельная категория `read_error`, а не `missing_line`: это
  техническая ошибка чтения, а не содержательное несовпадение; в тексте нарушения —
  пометка `(ОШИБКА)` (симметрично `normalizer`/`syncher`) и текст исключения.
- **default_rule**: применяется к видимым файлам группы без записи `group.file` и
  не начинающимся с `exclude_prefix`; если задано хотя бы одно из
  `extensions`/`exclude_extensions` — дополнительно отфильтрованным по расширению
  (`Path.suffix.lower()`, комбинация «И»; отброшенные файлы не читаются и не входят
  в `files_checked`).
- **`strict=false` (дефолт)**: после проверки прямых детей группы обход не
  спускается в её подпапки (`dirnames[:] = []` в терминах `os.walk`) — они не
  классифицируются ни группой, ни тематическим узлом. F14 (пустая группа) не
  затрагивается — это отдельный рекурсивный обход, не связанный с основным циклом
  `os.walk`.
- Итог — список нарушений (дедуп+сорт по пути и типу) и счётчики (`groups_checked`,
  `files_checked`).

## Осознанные допущения

- **Групповая папка внутри групповой папки** не запрещена и не имеет специальной
  обработки: она распознаётся как самостоятельная группа по тому же basename-правилу
  и проверяется собственными правилами независимо от контейнера; конфликта с
  проверкой loose-файлов нет (та проверяет только файлы, не подпапки). Внутри
  группы с `strict=false` (дефолт) это правило не действует — обход туда не
  заходит вовсе, поэтому вложенная папка с именем другой группы там не
  распознаётся.
- **Регистр имён групп и файлов** — литеральное сравнение, не gitignore-семантика.
- Класс движка и его инстанс именуются `FsSchemer`/`fssm` (см.
  [`naming-symmetry.md`](naming-symmetry.md)), по аналогии с `FsNormalizer`/`fsnm` и
  `FsChecker`/`fsch`.
- `apply_root`, указывающий сам на себя (`apply_root = "."`) или на каталог
  конфига через `..`-цепочку обратно к нему, не запрещён отдельно: это просто
  вырожденный случай текущего поведения (каталог проверки = каталог конфига),
  ничего не ломает.
