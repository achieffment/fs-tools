# Формат README (`## Обзор`)

> Claude-эквивалент [`.cursor/rules/readme-format.mdc`](../../.cursor/rules/readme-format.mdc). Применяется при работе с `README.md`.

Вводная часть `README.md` намеренно приведена к единому внутреннему формату
(см. коммит `de5bf85`). Правя вводную часть — сохраняй эту форму, а не
произвольный текст.

## Скелет вводной части

1. `# <имя-проекта>` — H1, без текста между заголовком и `## Обзор` (никакого
   «голого» вступительного абзаца перед секцией).
2. `## Обзор` — первая секция документа, ровно два блока внутри:
   - **Абзац-описание**: одно-два предложения вида «Проект —
     **<жирный обобщающий ярлык>** …», объясняющих архитектурную суть проекта
     на верхнем уровне (общее ядро, модель, из чего собран), БЕЗ перечисления
     конкретных режимов/правил — это задача следующего блока/таблицы.
   - **Список «как использовать»**: фраза `Репозиторий можно использовать
     как:` и буллет-лист из трёх пунктов формата `**<Роль>** — <описание>`.
     Канонические роли (порядок и точная формулировка могут отличаться по
     смыслу проекта, но набор фиксирован — все три пункта обязательны):
     1. **Готовый инструментарий** / **Готовый набор правил** — можно
        поставить/подключить как есть;
     2. **Референс архитектуры** — как пример решения конкретной инженерной
        задачи, не изобретая контракт с нуля;
     3. **Отправную точку** — скопировать структуру под свой набор и
        адаптировать/отключить готовое.
3. После списка ролей — по необходимости конкретика верхнего уровня (таблица
   режимов/правил проекта, короткая ссылка на единый конфиг/entry point и т.п.),
   но это уже не часть формулы «Обзора», а её продолжение.

## Дальше по документу (общий каркас, не идентичный контент)

После `## Обзор` README следует единому каркасу секций (порядок и состав
секций внутри каждой — по факту проекта, здесь фиксируется только форма):

- `## Требования` — версии/зависимости, буллет-лист.
- `## Установка` — название по смыслу проекта, не унифицируется.
- Секции по доменным единицам проекта (режимы CLI / custom-правила и т.п.) —
  каждая со своим форматом таблиц/примеров, специфичным для проекта.
- Секция разработки/тестирования (`## Разработка`).
- Опционально — секция перекрёстных ссылок на связанную документацию в конце.

## Что НЕ унифицируется

- Конкретные названия секций после «Обзора» (кроме самой формулы) — они
  отражают специфику проекта, а не общий канон.
- Таблицы форматируются по [`comments-style.md`](comments-style.md)
  (выравнивание столбцов GFM-таблиц) — эта правка не про формат таблиц, а про
  форму вводного текста.

## Проверка перед завершением правки README

- [ ] `## Обзор` идёт сразу после `# <имя-проекта>`, без текста между ними.
- [ ] Абзац-описание — не перечисление режимов/правил, а архитектурная суть.
- [ ] Список «как использовать» — ровно три пункта с жирными ролями
      (инструментарий/референс/отправная точка).
- [ ] Остальные секции README не потеряли контент при правке вводной части.
