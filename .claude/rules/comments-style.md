# Комментарии — кратко и по существу

> Claude-эквивалент [`.cursor/rules/comments-style.mdc`](../../.cursor/rules/comments-style.mdc). Применяется при работе с `**/*.py`, `**/*.md`, `**/*.toml`.

Объясняй неочевидное (зачем, ограничения, компромиссы), не пересказывай код.

## Код (Python, PEP 8)

- Inline-комментарий — минимум 2 пробела до `#`, затем `# ` и текст.
- Докстринги и комментарии — на русском, лаконично.

## Документация (все `*.md` проекта)

Inline-комментарии в **командных** fenced-код-блоках (`bash`, `sh`, `shell`,
`powershell`, `pwsh`, `bat`, `cmd`) держим на **одном уровне** (выровнены в
одну колонку) по правилу:
- выравнивание считается по локальному подблоку между пустыми строками;
- опорная длина берётся по **самой длинной непустой строке подблока** (кроме строк,
  начинающихся с `#` после отступов);
- в строке с inline-комментарием отступ до `#` — **ровно 4 пробела** от этой опорной длины;
- в остальных строках добавляй пробелы так, чтобы `#` стоял в той же колонке;
- если в подблоке меньше двух строк с inline-комментарием — выравнивание не требуется.

Проверка этого правила автоматизирована тестом
`tests/shared/test_markdown_comments.py` и считается обязательной частью
`pytest`-цикла.

Не выравнивай по случайно завышенному отступу; опорная колонка считается как
`длина_самого_длинного_текста + 4`. Это правило действует для всех
markdown-документов репозитория (`README.md`, `AGENTS.md`, `examples/*`, `docs/*`
и т.д.).

    pytest                          # тесты
    ruff check .                    # линтер
    mypy --strict -p fs_tools       # типы

## Таблицы (Markdown)

Столбцы GFM-таблиц во всех `*.md` проекта выровнены по ширине по правилу:
- ширина столбца — длина **самой длинной** ячейки этого столбца среди всех строк
  таблицы (заголовок и тело, разделительная строка не считается), но не меньше 3;
- содержимое ячейки дополняется пробелами справа до этой ширины; вокруг
  дополненного содержимого — по одному пробелу перед `|` и после (`| текст |`);
- разделительная строка — дефисы без пробелов, длиной `ширина + 2` (та же длина,
  что и у заполненной ячейки данных вместе с её пробелами-паддингами), без
  выравнивающих `:`;
- каждая таблица выравнивается независимо; строки таблицы должны начинаться и
  заканчиваться `|` (без него — не таблица для этого правила);
- экранированный `\|` внутри ячейки (напр. запрещённые на Windows символы в
  README) не считается разделителем столбцов.

Проверка этого правила автоматизирована тестом
`tests/shared/test_markdown_tables.py` и считается обязательной частью
`pytest`-цикла.

## Файлы `*.toml`

Подблок — тот же принцип, что и у Markdown (граница — пустая строка), но
профиль выравнивания у реальных `*.toml`-файлов репозитория
(`examples/schemer/.fs-sch.toml`, `examples/syncher/.fs-syn.toml`) — свой, а
не «то же самое с `.toml`-синтаксисом»:
- опорная колонка считается **только по строкам с inline-комментарием**
  внутри подблока, а не по всем его строкам — иначе далёкая длинная строка без
  комментария (`default_rule = { line = 1, text = "..." }`) утягивала бы
  выравнивание одиночного соседнего комментария;
- отступ до `#` — **ровно 2 пробела** от этой опорной длины (не 4, как у
  Markdown-fenced-блоков) — компактнее, ближе к обычному PEP 8-отступу;
- выравнивание обязательно уже при **одной** строке с inline-комментарием в
  подблоке (не только при ≥2, как у Markdown) — при единственной строке это и
  есть минимальный двухпробельный отступ, никакого исключения не нужно;
- `#` внутри TOML-строкового литерала (`text = "# Заметки"`) — не начало
  комментария, а литерал; граница ищется с учётом кавычек, а не регэкспом (в
  отличие от Markdown).

Проверка автоматизирована тестом `tests/shared/test_toml_comments.py` и
считается обязательной частью `pytest`-цикла. Общая механика флаша подблока
для обеих проверок (Markdown и TOML) вынесена в `tests/shared/comment_align.py`
(`AlignmentGroup(pad, min_rows, include_plain)`) — не дублируй её при
добавлении новых проверок выравнивания; профили (Markdown: `pad=4, min_rows=2,
include_plain=True`; TOML: `pad=2, min_rows=1, include_plain=False`) задаются
параметрами, а не копированием логики.

## Чего не делать

- Оставлять построчные комментарии, пересказывающие соседний код;
- Смешивать выровненную и невыровненную колонку `#` внутри одного подблока
  командного fenced-блока;
- Оставлять невыровненные столбцы GFM-таблиц после точечной правки строки
  (добавить/убрать ячейку) — перевыравнивать всю таблицу;
- Выравнивать по случайно завышенному отступу вместо `длина_самого_длинного_текста + 4`
  (Markdown) / `+ 2` (`*.toml`).
