# Фильтр путей `.fs-ignore` (gitignore-семантика)

> Claude-эквивалент [`.cursor/rules/path-matching.mdc`](../../.cursor/rules/path-matching.mdc). Применяется при работе с `src/fs_tools/normalizer/ignore.py`, `src/fs_tools/normalizer/engine.py`.

Фильтр (`fs_tools/normalizer/ignore.py`) — настоящая `.gitignore`-семантика поверх
`pathspec`. Инварианты приняты намеренно — НЕ «чинить» без явной задачи.

## Движок и формат

- Единый файл `.fs-ignore` лежит в нормализуемом каталоге (корне `apply`) и читается
  из него (`load_fs_ignore(root)` в `runner.py`, НЕ из корня проекта). Стиль
  `.gitignore`: обычная строка исключает, `!` — возвращает (override).
- Паттерны строятся через `pathspec.PathSpec.from_lines(_FACTORY, ...)`, где
  `_FACTORY` импортируется из `fs_tools/shared/pathspec_compat.py` и выбирается из
  доступных (`gitignore`, иначе `gitwildmatch`) — не привязывайся к версии pathspec.
- Файл НИКОГДА не изменяется (только чтение, `utf-8-sig`). Нет файла → `None`
  (фильтр выключен); пустой/из комментариев → `FsIgnore` без правил. Комментарий —
  только ведущий `#`.
- Рабочий демо-файл — `examples/normalizer/.fs-ignore` (секция `10-ignore`); его
  паттерны привязаны к `10-ignore`. Под git отслеживается.

## Семантика матчинга (gitignore)

- `*` — символы в пределах сегмента; `**` — ноль или более сегментов; `?` — один
  символ; `[abc]`/`[a-z]` — КЛАСС символов; завершающий `/` (`build/`) — только
  каталог и содержимое.
- **СМЕНА КОНТРАКТА:** `[abc]` и `?` — АКТИВНЫЕ метасимволы. Литеральные скобки/`?`
  экранируются `\` (`Файл \[1\]`). Разделитель — только `/`; `\` — экранирование.
- **Префикс `./` НЕ поддерживается** (`.` — обычный сегмент). Пиши `foo` (basename)
  или `/foo` (якорь).
- **`#` — только ведущий**: срединный `#` литерален (`C#`, `C++`).
- **Якоря**: паттерн без `/` (или только с завершающим `/`) — basename (любая
  глубина); паттерн с `/` внутри/в начале — привязан к корню нормализации.
- **Порядок строк значим**: решение принимает ПОСЛЕДНЯЯ совпавшая строка; `!`
  возвращает убранное ранее.
- **Регистронезависимость** (как git `core.ignorecase=true`): `Archive` == `archive`.
  Паттерны пересобираются с `re.IGNORECASE`. Мотив: `CaseRule` капитализирует
  родителей (`file-glob` → `File-glob`), и регистрозависимый якорь рвал бы
  идемпотентность фильтра.

## Относительность и обход

- Матчинг ОТНОСИТЕЛЬНО корня `apply`: `FsNormalizer` считает
  `path.relative_to(root)` и зовёт `FsIgnore.matches(rel, is_dir)`; каталогам
  добавляется завершающий `/`.
- `probe` (в `engine.py`) включается при `FsIgnore.has_overrides()` (есть правило-`!`):
  исключённые каталоги обходятся (внутри возможны возвращённые потомки). Без `!`
  исключённый каталог обрезается.
- Исключённые объекты не попадают в счётчики `renamed`/`skipped`.
- **Скрытые (на `.`) — раньше фильтра**: `_hidden` отсекает их ДО `.fs-ignore`,
  поэтому `!` их не возвращает. Сам `.fs-ignore` скрыт — пропускается обходом.

## Осознанные допущения

- `!` может вернуть потомка внутри исключённого каталога (оцениваем каждый путь
  независимо) — намеренно, под сценарий «исключить каталог, вернуть отдельные файлы».
- Относительная gitignore-модель: матчинг родителей выше выбранного каталога и токен
  диска (`D:` ↔ `/mnt/d`) не поддерживаются — не возвращай без явной задачи.
- Регистронезависимость лечит только разницу в РЕГИСТРЕ родителей. Не-ASCII имя
  родителя `TransliterationRule` меняет целиком (`Документы` → `Dokumenty`) — такой
  якорь рвётся на повторном прогоне (известное ограничение).
