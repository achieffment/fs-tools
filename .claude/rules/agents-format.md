# Формат AGENTS.md/CLAUDE.md и скелет файла-правила

> Claude-эквивалент [`.cursor/rules/agents-format.mdc`](../../.cursor/rules/agents-format.mdc). Применяется всегда.

Правило описывает форму, а не содержание: одинаковый порядок и наименование
разделов в справочниках агента и в парных файлах правил, чтобы открывший
`AGENTS.md`/`CLAUDE.md`/любое правило видел одну и ту же структуру
независимо от конкретного проекта. Контент внутри разделов остаётся
проектно-специфичным.

## Скелет `AGENTS.md`

1. `# AGENTS.md` — заголовок ровно с этим текстом.
2. `## Обзор` — одно-два предложения: что за репозиторий, из чего собран.
3. `## Роль` — короткий персона-абзац: в чём агент эксперт применительно к
   этому репозиторию, требование точных правок с сохранением контрактов,
   минимальный diff, запрет переписывать с нуля или навязывать архитектуру
   без запроса.
4. Проектная секция раскладки/scope (название и форма — по факту проекта:
   дерево каталогов, таблица «область → файлы» и т.п.) — обязательна, но не
   унифицируется по имени раздела.
5. `## Правила проекта` — сводная таблица `Cursor | Claude | Тема` со всеми
   парными правилами (тот же список, что в карте `rules-sync.md`/`.mdc`, —
   поддерживать синхронно при добавлении/удалении правила).
6. `## Рабочий процесс` — нумерованные шаги типового цикла правки: прочитать
   релевантное правило/тесты → design-check (SRP/DRY, нет ли уже похожей
   утилиты) → минимальный diff → соответствие локальным соглашениям
   (naming-symmetry и т.п.) → сохранение контрактов (коды возврата,
   терминальный вывод, текст веб-хука и т.п.) → регистрация нового
   компонента в точке сборки (если применимо) → полный цикл проверок →
   синхронизация docs по матрице `docs-consistency.md`.
7. Проектные технические секции (команды проверки, ключевые принципы,
   раскладка тестов, ограничения и т.п.) — состав и порядок по факту
   проекта, не унифицируются.
8. `## Границы` — короткая формулировка (не угадывать, не коммитить/push без
   просьбы, не раздувать scope, не трогать сторонние пакеты, не
   переформатировать вне задачи) со ссылкой на детальное правило, если оно
   вынесено отдельно (см. «Правило дублирования» ниже).
9. `## Коммуникация` — короткая норма (язык пользователя/репозитория,
   кратко: что/зачем/как проверено).
10. Секция аудита изменений и проекта — форма по факту проекта: инлайновый
    чеклист **или** (предпочтительно) ссылка на выделенный skill/rule, если
    контракт аудита достаточно велик, чтобы его дублирование в `AGENTS.md`
    создавало риск рассинхронизации.

## Скелет `CLAUDE.md`

1. Заголовок `# CLAUDE.md`.
2. Интро-строка: «Справочник и правила проекта для Claude Code — эквивалент
   `.cursor/rules/` для Cursor. Основной AI-справочник — @AGENTS.md.»
3. `## Правила (Claude)` — список всех файлов `.claude/rules/*.md` с кратким
   описанием каждого (не таблица — маркированный список), **в алфавитном
   порядке файлов** — том же порядке, что в карте `rules-sync.md` и в таблице
   «Правила проекта» `AGENTS.md` (три листинга одного набора правил должны
   идти в одинаковом порядке, а не только содержать один и тот же набор).
4. `## Синхронизация с Cursor` — короткий абзац о едином каноне и ссылка на
   `rules-sync.md`.

## Скелет файла-правила (`.md`/`.mdc` пара)

1. Заголовок `# <Название правила>`.
2. У Claude-версии — blockquote-строка
   `> Claude-эквивалент [.mdc-файл](путь). Применяется <когда>.`; у
   Cursor-версии — тот же смысл в YAML frontmatter (`description`,
   `alwaysApply`/`globs`).
3. Интро-абзац — зачем правило существует (проблема/инвариант, который оно
   защищает), не пересказ политики.
4. Разделы политики — по смыслу правила, разбиение и заголовки свободны.
5. Для правил с высоким риском ошибки при неполном применении (форматы,
   контракты, чувствительные к порядку операции) — финальный раздел-чеклист
   `## Проверка перед завершением` (или аналогичное название).

## Правило дублирования контента

Между `AGENTS.md`/`CLAUDE.md` и соответствующим правилом допустимо
дублировать **сводные таблицы/списки** (например, «Правила проекта» в
`AGENTS.md` и карта в `rules-sync.md` — обе нужны для разных целей: чтение
vs синхронизация). Недопустимо дублировать **детальную политику** — вместо
этого `AGENTS.md`/README ссылаются на правило (см.
[docs-consistency.md](docs-consistency.md), раздел «Антипаттерны»).

## Применение

Это правило — эталон, к которому приводится каждый проект отдельно; при
переносе эталона в другой проект допускаются механические различия
(имена проекта, состав проектных секций), но не различия в наборе/порядке
разделов, перечисленных выше.
