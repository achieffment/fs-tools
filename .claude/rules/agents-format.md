# Формат AGENTS.md/CLAUDE.md и скелет файла-правила

> Claude-эквивалент [`.cursor/rules/agents-format.mdc`](../../.cursor/rules/agents-format.mdc). Применяется всегда.

Правило описывает форму, а не содержание: одинаковый порядок и наименование
разделов в справочниках агента и в парных файлах правил, чтобы открывший
`AGENTS.md`/`CLAUDE.md`/любое правило видел одну и ту же структуру
независимо от конкретного проекта. Контент внутри разделов остаётся
проектно-специфичным.

## Скелет `AGENTS.md`/`CLAUDE.md`

`AGENTS.md` и `CLAUDE.md` — самодостаточная синхронизированная пара: каждый
файл полностью описывает репозиторий сам по себе (не отсылает читателя ко
второму файлу за содержанием), а различает их только целевой инструмент
(Cursor-агностичный `AGENTS.md` vs Claude Code `CLAUDE.md`). Скелет ниже —
единый для обоих файлов; единственные допустимые различия между ними
отмечены явно в пунктах 1, 5 и 6 — во всём остальном состав, порядок и
содержание разделов идентичны (см. «Правило дублирования контента» ниже).

1. Заголовок: ровно `# AGENTS.md` в `AGENTS.md`, ровно `# CLAUDE.md` в
   `CLAUDE.md`. В `AGENTS.md` сразу после заголовка идёт `## Обзор` (без
   вступительного текста); в `CLAUDE.md` между заголовком и `## Обзор` —
   ровно одна интро-строка: «Справочник и правила проекта для Claude Code —
   эквивалент `.cursor/rules/` для Cursor.»
2. `## Обзор` — одно-два предложения: что за репозиторий, из чего собран.
3. `## Роль` — короткий персона-абзац: в чём агент эксперт применительно к
   этому репозиторию, требование точных правок с сохранением контрактов,
   минимальный diff, запрет переписывать с нуля или навязывать архитектуру
   без запроса.
4. Проектная секция раскладки/scope (название и форма — по факту проекта:
   дерево каталогов, таблица «область → файлы» и т.п.) — обязательна, но не
   унифицируется по имени раздела.
5. `## Правила проекта` — сводная опись всех парных правил (тот же список,
   что в карте `rules-sync.md`/`.mdc`, — поддерживать синхронно при
   добавлении/удалении правила). Единственное форматное отличие пары: в
   `AGENTS.md` — таблица `Cursor | Claude | Тема`; в `CLAUDE.md` —
   маркированный список файлов `.claude/rules/*.md` с кратким описанием
   каждого (Cursor-колонка избыточна в справочнике для Claude Code). Оба
   перечисляют один и тот же набор правил **в одном и том же алфавитном
   порядке** (том же, что в карте `rules-sync.md` — три листинга одного
   набора правил должны идти в одинаковом порядке, а не только содержать
   один и тот же набор).
6. Заголовок секции синхронизации называет **парный файл**, а не самого
   себя: `## Синхронизация с Claude` в `AGENTS.md` (описывает пару с
   `CLAUDE.md`/`.claude/rules/`), `## Синхронизация с Cursor` в `CLAUDE.md`
   (описывает пару с `.cursor/rules/`) — «AGENTS» и «CURSOR» не одно и то же,
   называть секцию своим же именем некорректно. Содержание — короткий абзац о
   едином каноне (Cursor `.mdc` + Claude `.md` несут один и тот же канон
   правил) с явным указанием двусторонности (Cursor → Claude и Claude →
   Cursor — правка в любом наборе переносится в парный в этом же шаге) и
   ссылка на `rules-sync.md`. Идёт сразу после «Правила проекта» в обоих
   файлах — тематически продолжает её.
7. `## Рабочий процесс` — нумерованные шаги типового цикла правки: прочитать
   релевантное правило/тесты → design-check (SRP/DRY, нет ли уже похожей
   утилиты) → минимальный diff → соответствие локальным соглашениям
   (naming-symmetry и т.п.) → сохранение контрактов (коды возврата,
   терминальный вывод, текст веб-хука и т.п.) → регистрация нового
   компонента в точке сборки (если применимо) → полный цикл проверок →
   синхронизация docs по матрице `docs-consistency.md`.
8. Проектные технические секции (команды проверки, ключевые принципы,
   раскладка тестов, ограничения и т.п.) — состав и порядок по факту
   проекта, не унифицируются.
9. `## Границы` — короткая формулировка (не угадывать, не коммитить/push без
   просьбы, не раздувать scope, не трогать сторонние пакеты, не
   переформатировать вне задачи) со ссылкой на детальное правило, если оно
   вынесено отдельно (см. «Правило дублирования» ниже).
10. `## Коммуникация` — короткая норма (язык пользователя/репозитория,
    кратко: что/зачем/как проверено).
11. Секция аудита изменений и проекта — форма по факту проекта: инлайновый
    чеклист **или** (предпочтительно) ссылка на выделенный skill/rule, если
    контракт аудита достаточно велик, чтобы его дублирование создавало риск
    рассинхронизации.

## Скелет файла-правила (`.md`/`.mdc` пара)

1. Заголовок `# <Название правила>`.
2. У Claude-версии — blockquote-строка
   `> Claude-эквивалент [.mdc-файл](путь). Применяется <когда>.`; у
   Cursor-версии — тот же смысл в YAML frontmatter (`description`,
   `alwaysApply`/`globs`).
3. Интро-абзац — зачем правило существует (проблема/инвариант, который оно
   защищает), не пересказ политики.
4. Разделы политики — по смыслу правила, разбиение и заголовки свободны.
5. Для правил с высоким риском ошибки при неполном применении (форматы,
   контракты, чувствительные к порядку операции) — финальный раздел-чеклист
   `## Проверка перед завершением` (или аналогичное название).

## Правило дублирования контента

`AGENTS.md` и `CLAUDE.md` — намеренно синхронизированная пара (по аналогии с
парами `.cursor/rules/*.mdc`/`.claude/rules/*.md`): полное содержательное
дублирование разделов между ними — это canon, а не нарушение DRY. Правя один
файл по существу (не только форматную деталь из п. 1/5/6 «Скелета
`AGENTS.md`/`CLAUDE.md`» выше) — в этом же шаге переносить правку в парный
файл (см. также [`rules-sync.md`](rules-sync.md), который описывает тот же
принцип для пар правил).

Между `AGENTS.md`/`CLAUDE.md` и соответствующим **правилом** (`.claude/rules/*.md`)
допустимо дублировать **сводные таблицы/списки** (например, «Правила проекта» в
`AGENTS.md` и карта в `rules-sync.md` — обе нужны для разных целей: чтение
vs синхронизация). Недопустимо дублировать **детальную политику правила** —
вместо этого `AGENTS.md`/`CLAUDE.md`/README ссылаются на правило (см.
[docs-consistency.md](docs-consistency.md), раздел «Антипаттерны»).

## Применение

Это правило — эталон, к которому приводится каждый проект отдельно; при
переносе эталона в другой проект допускаются механические различия
(имена проекта, состав проектных секций), но не различия в наборе/порядке
разделов, перечисленных выше.
