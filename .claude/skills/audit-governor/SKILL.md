# Audit Governor

## Режимы запуска

Используй только один из двух режимов:

- `audit changed` — аудит внесённых правок.
- `audit full` — полный аудит проекта.

Если режим не указан, по умолчанию используй `audit changed`.

## Обязательные источники истины

Перед аудитом прочитай:

- `AGENTS.md`
- `README.md`
- `examples/README.md`
- `examples/checker/README.md`
- `examples/normalizer/README.md`
- `examples/syncher/README.md`
- `examples/schemer/README.md`
- `pyproject.toml`
- `CLAUDE.md` (корневой) и все `.claude/rules/*.md`
- `checklist.md` этого skill

## Область проверки

### Для `audit changed`

Проверяй:

- unstaged и staged изменения;
- untracked файлы;
- все коммиты текущей ветки относительно реальной base-ветки (`main` или
  `master`);
- все связанные изменения в коде, тестах, examples, документации, правилах,
  агентных инструкциях.
- не ограничивайся только последним коммитом.

### Для `audit full`

Проверяй весь репозиторий целиком по тем же критериям качества и
консистентности.

## Что обязательно валидировать

- абсолютный coverage всех пластов: код, тесты, examples, документация,
  комментарии, правила проекта (`CLAUDE.md`, `.claude/rules/*.md`), агентные
  инструкции (`AGENTS.md`);
- высокий инженерный уровень реализации: корректная декомпозиция, SRP, DRY,
  минимализм, простота и читаемость;
- консистентность между кодом, тестами, examples, документацией и
  правилами;
- локальные соглашения именования и симметрии
  (`.claude/rules/naming-symmetry.md`), включая пары для
  `fs_tools/cli.py` и `normalizer/engine.py`;
- комментарии: минимальность, актуальность, корректность, выравнивание в
  Markdown-код-блоках и в файлах `*.toml` по `.claude/rules/comments-style.md`;
- кроссплатформенность и безопасность по правилам проекта;
- отсутствие необоснованных suppression-комментариев (`# pylint: disable`,
  `# noqa`, `# type: ignore`);
- отсутствие комментариев-пропусков правил без явной необходимости и
  документированного обоснования;
- dry-run контракт: у `normalizer` и `syncher` при `--dry-run` пишется
  `.fs-log` с меткой режима `dry-run` и планом изменений;
- контракт логирования `.fs-log` для всех утилит: единый заголовок блока
  (`дата`, `Инструмент`, `Режим`, `Результат`), append-only поведение и
  режимные маркеры пустого результата (`(изменений нет)` /
  `(нарушений нет)`) должны оставаться консистентными между `shared.log`,
  режимными `log.py`, тестами и документацией;
- runner-паттерн: режимы с собственными флагами используют
  `_build_parser()` и не плодят одноразовые константы для `path_help`;
- контракт терминального вывода: финальный блок в
  `normalizer`/`checker`/`syncher`/`schemer` должен оставаться двухстрочным
  (`Статус:` + `Сводка:`) и синхронизированным в коде, тестах и
  документации (у `schemer` статус — только `ok`/`error`, без `warn`);
- контракт текста веб-хуков: фиксированные строки
  `fs-checker - выполнен с ошибкой.`, `fs-syncher - выполнен с ошибкой.` и
  `fs-schemer - выполнен с ошибкой.` плюс условия отправки (checker: при
  missing или ошибках сканирования `**`-обхода (`errlist`); syncher: только
  production и наихудший код `2`/`3`; schemer: только при наличии
  нарушений) должны быть синхронизированы между раннерами, тестами и docs.

Если для прохождения проверок нужны изменения в `CLAUDE.md`, `.claude/rules/*.md`
или `AGENTS.md`, вноси минимальные согласованные правки и повторяй цикл проверок.

## Обязательный цикл проверок

Используй команды из `checklist.md` и добивайся полного green:

- `.venv/bin/python -m pytest -q`
- `.venv/bin/python -m pylint --persistent=n --recursive=y src tests/*`
- `.venv/bin/python -m ruff check .`
- `.venv/bin/python -m mypy --strict -p fs_tools`

После каждой серии исправлений запускай проверки повторно до полного green.
После собственных исправлений выполняй повторный аудит тем же режимом
(`audit changed` или `audit full`) до полного green.

Отдельно контролируй, что в `pytest` проходят `tests/shared/test_markdown_comments.py`
и `tests/shared/test_toml_comments.py` (автопроверка выравнивания
inline-комментариев в командных fenced-блоках Markdown и в файлах `*.toml`; оба
считают по локальным подблокам между пустыми строками, но с разными профилями
опорной колонки — точные параметры в `.claude/rules/comments-style.md`, не
дублируй их здесь).

Если есть расхождение между IDE-диагностикой и общим прогоном `pylint`,
выполняй точечный запуск по проблемному файлу:

- `.venv/bin/python -m pylint --persistent=n tests/path/to/file.py`

## Политика исправлений

- Исправляй автоматически всё, что можно исправить корректно и безопасно.
- Не вноси косметические изменения без пользы.
- Не ломай публичные контракты без крайней необходимости.
- Не ослабляй безопасность и кросс-платформенные гарантии.
- Не оставляй частичную проверку: завершай аудит только после валидации
  всех пластов и повторного цикла проверок.
- Если выявлено, что правила проекта или `AGENTS.md` недостаточны для
  прохождения проверок, обновляй их минимально и согласованно с фактическим
  поведением кода.

## Формат финального ответа

Возвращай кратко:

1. список изменений по файлам;
2. результаты обязательных проверок;
3. что исправлено для консистентности;
4. подтверждение DoD.
