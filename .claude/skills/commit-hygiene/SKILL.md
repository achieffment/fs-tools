# Гигиена коммитов: секреты и стилистика сообщений

Перед `git commit`/`git push` (и по прямому запросу «проверь на секреты» /
«оформи коммит») выполнять обе проверки ниже — они относятся к одному моменту
в рабочем процессе и разъезжаться не должны.

## 1. Секреты и конфиденциальные данные

Разово вставленный `.env` или ключ в истории остаётся там навсегда даже после
удаления файла — дешевле проверить до коммита, чем чистить историю после.

### Что проверять

- **Staged/untracked файлы** (`git status`, `git diff --cached`): не должно быть
  реальных `.env`, ключей (`*.pem`, `*.key`, `id_rsa`, `id_ed25519`,
  `credentials.json`), файлов с `secret`/`token`/`credential` в имени — если это не
  заведомо безопасная фикстура (см. ниже).
- **Содержимое диффа** — паттерны реальных секретов: `AKIA[0-9A-Z]{16}` (AWS),
  `xox[baprs]-...` (Slack), `ghp_`/`gho_`/`github_pat_` (GitHub), `AIza...`
  (Google), `sk-...` (общие API-ключи), `-----BEGIN ... PRIVATE KEY-----`, а также
  буквальные присвоения `password = "..."`, `secret = "..."`, `api_key = "..."` с
  непустым значением, не являющимся плейсхолдером.
- **Вся история при подозрении/по явному запросу** (не только текущий дифф):
  `git log --all -p` теми же паттернами — если секрет уже закоммичен в прошлом,
  текущий чистый дифф этого не покажет.

### Как отличить фикстуру от секрета

Плейсхолдеры — не нарушение: `.env.example` с `https://example.com/hook` и
`секретный-токен`; демо-фикстура `examples/normalizer/08-edge-cases/hidden/.env`
(содержимое — заглушка `fixture: .env`, участвует в тесте пропуска скрытых
файлов). Отличать по содержимому, а не только по имени файла: `.env` с реальным
`https://`-URL и непустым токеном — уже не фикстура.

### Что делать при обнаружении

- Файл/строка ещё не закоммичены → убрать из staging, добавить паттерн в
  `.gitignore` при необходимости.
- Секрет уже есть в истории (даже в старом коммите) → не молчать: сообщить
  пользователю и не выполнять `push` без его решения — переписывание истории
  (`filter-repo`/`BFG`, force-push) — необратимое действие, требующее явного
  подтверждения (см. общее правило о рискованных действиях).

### Команда для быстрой проверки (POSIX shell)

    git diff --cached | grep -inE 'AKIA[0-9A-Z]{16}|xox[baprs]-|ghp_[A-Za-z0-9]{36}|-----BEGIN (RSA|DSA|EC|OPENSSH|PRIVATE) KEY-----|AIza[0-9A-Za-z_-]{35}|sk-[A-Za-z0-9]{20,}'

## 2. Стилистика сообщений (Conventional Commits, как в текущей истории)

Заголовок — по факту сложившийся в `git log` этого репозитория формат, не
собственная выдумка на момент коммита:

    <type>(<scope>): <краткое summary в повелительном наклонении, англ., без точки>

- `type` — один из фактически используемых в истории: `feat`, `fix`, `refactor`,
  `docs`, `chore`, `test`, `style`, `build`. Не вводи новый тип без необходимости
  (`perf`, `ci` и т.п.) — сверься с `git log --oneline`, вдруг подходящий уже есть.
- `scope` — затронутый модуль/пакет режима (`schemer`, `checker`, `normalizer`,
  `syncher`, `shared`, `rules`, `claude`, `bin`, `audit`, `readme`, `logging`,
  `report`, `offload` и т.п.); при затрагивании нескольких — через запятую
  (`feat(schemer,checker,normalizer): ...`). Скоп опускается только когда правка
  сквозная и не привязана к одному модулю (`docs: fix stale rule references...`,
  `refactor: align mode artifact filenames...`).
- Summary — короткое (обычно < 72 символов), в повелительном наклонении
  (`add`, `fix`, `move`, `rename`, а не `added`/`fixes`), на английском.

Тело сообщения (не всегда нужно — короткая механическая правка может обойтись
без него):

- Фокус на **почему**, а не на пересказе диффа построчно (дифф и так виден).
- Может быть на русском или английском — в истории репозитория встречаются оба
  варианта; выбирай тот, что естественнее для конкретного объяснения.
- Заворачивай строки на разумную ширину (как остальной текст в репозитории).
- Заканчивается трейлером `Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>`,
  когда сообщение сформировано ассистентом (см. общий раздел «Committing changes
  with git» в системных инструкциях) — отдельной пустой строкой перед трейлером.

### Примеры из фактической истории (ориентир, не текст для копирования)

    chore(audit): fix stale description and rules-sync drift found in full audit
    refactor(shared): rename journal file .fs-log to .fs-log.log
    feat(schemer): filter default_rule content-check by file extension
    docs(rules): sync naming-symmetry cross-link in Cursor scheme-format rule

## Порядок действий при подготовке коммита

1. Прогнать проверку секретов (раздел 1) на staged-изменениях.
2. Сформировать сообщение по формату раздела 2, опираясь на `git log --oneline -20`
   для сверки стиля, если сомневаешься в типе/скопе.
3. Коммитить только по явному запросу пользователя (см. общее правило — не
   коммитить проактивно).
