# fs-normalizer

CLI-утилита для рекурсивной нормализации имён файлов и папок: транслитерация в ASCII, приведение дат к ISO-формату, единый стиль разделителей и регистра. Поведение идемпотентно — повторный запуск над уже нормализованным деревом ничего не меняет.

## Возможности

- Рекурсивный обход каталога с переименованием файлов и папок.
- Транслитерация не-ASCII символов (кириллица, умляуты, emoji и т.п.) в ASCII. Мягкий и твёрдый знаки (`ь`/`ъ`) удаляются, а не превращаются в апостроф; запрещённые в именах файлов Windows символы (`< > : " | ? *`), которые может породить транслитерация типографики (`«»`→`<<`/`>>`), вырезаются.
- Обработка круглых и квадратных скобок: с числом (дубли файловых менеджеров `Файл (1)`, `Файл [1]`) — скобки вырезаются, число дополняется ведущим нулём (→ `fail-01`); с текстом (`инн (Нового договора нет)`) — скобки сохраняются. Непарные/несовпадающие скобки (`Файл (1`, `инн (текст]`) вырезаются как невалидные.
- Распознавание дат в разных форматах и приведение к ISO (`YYYY-MM-DD`) с плейсхолдерами `00` для недостающих компонентов.
- Ведущий ноль для однозначных числовых токенов (`1_file` → `01_file`).
- Пробелы → дефис; цепочки дефисов вокруг пробелов схлопываются в один (`Резюме - подготовка` → `reziume-podgotovka`), но намеренные дефисы без пробелов сохраняются (`file--improved`). Обрезка «мусора» по краям имени (ведущий `_` сохраняется и у файлов, и у папок, на краю сохраняется парная круглая скобка), единый регистр (папки — с заглавной, файлы — в нижнем; общепринятые имена вроде `README` сохраняют регистр; у папок ведущий `_` сохраняется, а капитализируется первая буква после него — `_private` → `_Private`).
- Расширение файла не изменяется.
- Скрытые файлы и папки (начинающиеся с `.`) пропускаются и не обходятся.
- Список исключений (`exclude.txt` в корне проекта): объекты, чей путь содержит указанный фрагмент, не нормализуются, внутрь исключённых каталогов утилита не заходит, и такие объекты не учитываются в подсчёте (см. «Исключения»).
- Повторное включение (`include.txt`): переопределяет `exclude.txt` и возвращает выбранные пути (с поддержкой glob: `*` внутри сегмента, `**` — целый сегмент) к нормализации (см. «Повторное включение»).
- Корневой каталог не переименовывается.
- Безопасная обработка конфликтов: если целевое имя уже занято другим объектом, переименование пропускается (данные не перезаписываются).
- Имя всегда остаётся одним компонентом пути: разделители (`/`, `\`) и управляющие символы, которые может ввести транслитерация, заменяются на `-` — объект нельзя случайно переместить в другой каталог или потерять.
- Выбор каталога: нативный проводник Windows (на Windows и в WSL), стандартный диалог macOS либо ввод пути в терминале (на обычном Linux).

## Требования

- Python 3.10+ (используется синтаксис `X | None`).
- [Unidecode](https://pypi.org/project/Unidecode/) — единственная рантайм-зависимость.

Стандартной библиотеки достаточно для выбора папки: на Windows и в WSL вызывается `powershell.exe` (нативный диалог `IFileOpenDialog`), на macOS — `osascript` (AppleScript `choose folder`), на обычном Linux — ввод пути в терминале. Дополнительных GUI-пакетов не требуется.

## Установка

Через обёртки установка не нужна: при первом запуске они сами создают `.venv` и ставят зависимости (требуется интернет один раз). Этот раздел — для прямого запуска `python3 normalize_fs.py` или разработки:

```bash
cd fs-normalizer
python3 -m venv .venv
source .venv/bin/activate    # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## Использование

Проще всего — через обёртку для своей ОС. При первом запуске она автоматически подготовит окружение (`.venv` + зависимости), затем запустит утилиту:

```bash
./normalize.bat    # Windows
./normalize.sh     # Linux/macOS (терминал)
```

На macOS для запуска двойным кликом в Finder используйте `normalize.command` (один раз сделайте его исполняемым: `chmod +x normalize.command`).

Альтернатива — прямой запуск (требует заранее подготовленного окружения, см. «Установка»):

```bash
python3 normalize_fs.py
```

Аргументы не нужны — каталог выбирается интерактивно. Поведение зависит от среды (в скобках — папка, предлагаемая по умолчанию):

- **Windows** — открывается нативный диалог выбора папки (`IFileOpenDialog`, `FOS_PICKFOLDERS`). Он сразу открывается на папке по умолчанию, при этом навигация по всем путям остаётся свободной. По умолчанию это рабочий каталог: при запуске из командной строки — каталог вызова, а при запуске через ярлык/двойной клик по `.bat` — его рабочая папка (для двойного клика она автоматически равна каталогу проекта; для ярлыка `.lnk` задайте поле «Рабочая папка» = каталог проекта).
- **WSL** — открывается тот же нативный диалог Windows, сразу на каталоге вызова (через UNC-путь `\\wsl.localhost\...`); выбранный путь автоматически переводится обратно в путь WSL. Навигация по всем путям (включая узел «Linux» и сеть) свободна.
- **macOS** — открывается стандартный системный диалог `choose folder` (через `osascript`), возвращается POSIX-путь; по умолчанию предлагается каталог, из которого вызвана утилита (с откатом на каталог проекта).
- **Обычный Linux** — запрашивается ввод пути к каталогу в терминале; по умолчанию (пустой ввод) используется каталог, из которого вызвана утилита.

> **О диалоге Windows/WSL.** Используется системный `IFileOpenDialog` в режиме выбора папки: `SetFolder` вместе с уникальным `SetClientGuid` жёстко открывает диалог на нужной папке (игнорируя запомненную последнюю папку — MRU) и принимает UNC-путь WSL, сохраняя свободную навигацию. Стандартные `OpenFileDialog`/`BrowseForFolder` этого не дают: первый игнорирует стартовую папку для UNC (открывает «Документы»), второй не принимает UNC как корень. Флаг `FOS_FORCEFILESYSTEM` намеренно **не** ставится: корневой узел «Linux» (`\\wsl.localhost`) не считается элементом файловой системы, и с этим флагом он исчез бы из дерева — то есть из-под Windows нельзя было бы добраться до папок WSL. Из WSL путь к Windows-процессу прокидывается через переменную `WSLENV` (иначе пользовательские переменные окружения не доходят до `powershell.exe`). Небольшой блок интеропа на C# вынесен в отдельный файл `normalizer/pick_folder.ps1` и исполняется через `powershell.exe`.

После завершения выводится сводка: сколько объектов переименовано и сколько пропущено.

### Быстрый запуск через алиас

Чтобы не указывать полный путь к обёртке каждый раз, добавьте алиас в конфиг своей оболочки.

bash/zsh — в `~/.bashrc` или `~/.zshrc`:

```bash
alias fsnorm="$HOME/Projects/Work/fs-normalizer/normalize.sh"
```

Примените изменения (`source ~/.bashrc`) — и запускайте командой `fsnorm`.

PowerShell — в файле профиля (`$PROFILE`). Используется функция-обёртка, чтобы корректно пробрасывать аргументы:

```powershell
function fsnorm { & "C:\path\to\fs-normalizer\normalize.bat" @args }
```

Перечитайте профиль (`. $PROFILE`) — и запускайте командой `fsnorm`.

## Исключения (`exclude.txt`)

Чтобы исключить отдельные каталоги или файлы из нормализации, перечислите их в файле `exclude.txt` в корне проекта (рядом с `normalize_fs.py`). Файл создаётся автоматически (пустым) при первом запуске обёртки и добавлен в `.gitignore`, поэтому у каждого пользователя свой локальный список.

В каждой строке — один фрагмент пути (каталог или имя файла). Пустые строки игнорируются. Примеры:

```text
Archive
Resources/Fonts
Home/Components
Home/Activities/Python/Projects
D:\Programs
notes.txt
*.bak
Projects/*/build
Projects/**/Data
```

Как это работает:

- **Поиск вхождений по сегментам пути.** Объект пропускается, если его полный путь содержит указанный фрагмент как непрерывную цепочку сегментов. `Archive` совпадёт с каталогом `.../Archive/...`, но не с `MyArchive` или `Archive2`; `Home/Components` совпадёт с `.../Home/Components/...`, но не с `.../Home/Other/Components`.
- **Работает и для файлов.** Одиночное имя `notes.txt` исключит файл с таким именем в любом месте дерева; уточнённый путь `Home/Documents/notes.txt` — только в этой цепочке каталогов. Совпадение по границе сегмента: `mynotes.txt` и `notes.txt.bak` не затрагиваются.
- **Проверяется полный путь**, включая родителей выше выбранного каталога: `Home/Components` защитит всё под `/home/.../Home/Components` независимо от того, какой каталог выбран для нормализации.
- **Терпимость к слешам** (сам файл при этом не изменяется): завершающий и ведущий слеш, а также случайные двойные/тройные слеши игнорируются — `Home/Components/`, `/Home/Components` и `Home//Components` равнозначны.
- **Одинаково на Windows, WSL, Linux и macOS.** Разделители `\` и `/` равнозначны, регистр не учитывается. Буква диска и WSL-монтирование сводятся к одному виду: `D:\` и `/mnt/d/` означают один и тот же диск `d`. Поэтому `D:\Programs` и `/mnt/d/Programs` совпадают в обе стороны, но **разные диски не путаются** — `E:\Programs` не исключит `D:\Programs`. Фрагмент без диска (`Programs`) совпадает на любом диске.
- **Исключённое не обходится и не считается.** Внутрь исключённого каталога утилита не заходит, его содержимое не трогается, а сами пропущенные объекты не попадают в итоговый счётчик (ни в «переименовано», ни в «пропущено»).
- **Glob-шаблоны (спецсимвол только `*`).** Звёздочка `*` означает любое количество символов (в том числе ноль) **внутри одного сегмента** — она не пересекает разделитель: `*.txt` (любой `*.txt` в этом сегменте), `a*` (сегмент на `a`), `*foo*` (содержит `foo`), `data-*-final`. Одиночная `*` — это ровно один сегмент целиком (`Projects/*/build`). Последовательность `**` **как целый сегмент** означает ноль или более сегментов (`Projects/**/Data`); в любой другой комбинации со звёздами в сегменте (`a**b`, `***`) лишние звёзды схлопываются в обычный intra-segment `*`. Символы `?`, `[`, `]` — **литералы**, а не метасимволы glob: паттерн `Файл [1]` совпадает буквально с сегментом `Файл [1]` (имена со скобками в проекте встречаются повсюду). Те же шаблоны работают и в `include.txt`.
- **Без файла — обычное поведение.** Если `exclude.txt` отсутствует или пуст, никакие проверки не выполняются.

> Осознанное допущение: диск с буквой сводится к одному сегменту-токену (`D:\` → `d`), поэтому путь с диском теоретически может совпасть и с обычным каталогом из одной буквы `d` в середине дерева. На практике такие имена не встречаются, а вариант точнее прежнего «полного отбрасывания диска».

## Повторное включение (`include.txt`)

`include.txt` — это **переопределение** `exclude.txt`: он возвращает к нормализации то, что убрал `exclude.txt`. Файл создаётся автоматически (пустым), добавлен в `.gitignore`, формат и канонизация путей те же (сегменты, терпимость к слешам, единый вид диска/`/mnt/<буква>`, glob: `*` внутри сегмента, `**` — целый сегмент, а `?`/`[`/`]` — литералы). Без `exclude.txt` (или если путь и так не исключён) `include.txt` ничего не меняет.

Пример: исключаем архивы и черновики проектов, но возвращаем папки `Data` внутри них.

```text
# exclude.txt
Archive
Home/Activities/Web/Projects

# include.txt
Activities/Web/Projects/**/Data
```

Как разрешается конфликт:

- **Совпадение include возвращает объект и его поддерево.** Если include совпал на каталоге `Data`, то и `Data`, и всё внутри снова нормализуется.
- **Глубже — важнее (ничья — за include).** Конфликт решается по глубине последнего совпавшего сегмента: если include совпал не мельче exclude — нормализуем, иначе пропускаем. При равной глубине побеждает include.
- **Следствие — повторное исключение вложенного.** Если внутри возвращённого `Data` снова попадается `Archive`, его exclude-совпадение глубже, поэтому он опять исключается. Чтобы вернуть и его, добавьте более глубокий include, например `Activities/Web/Projects/**/Data/**/Archive`.
- **Одинаково на всех ОС** и терпимо к слешам — как и у `exclude.txt`.

## Правила нормализации

Правила применяются к имени (без расширения для файлов) строго в указанном порядке:

| # | Правило | Что делает |
|---|---------|------------|
| 1 | `TransliterationRule` | Любой не-ASCII символ → ASCII (мягкий/твёрдый знаки `ь`/`ъ` удаляются, а не дают апостроф); разделители пути (`/`, `\`) и управляющие символы, которые может породить транслитерация (`½`→`1/2`, `∖`→`\`, U+2028→перевод строки), заменяются на `-`, чтобы имя осталось одним компонентом пути; запрещённые на Windows символы (`< > : " | ? *`, например из `«»`→`<<`/`>>`) вырезаются |
| 2 | `BracketsRule` | Круглые и квадратные скобки с числом/датой (без букв) → убираются (`(1)`/`[1]`→`1`, дальше ведущий ноль); скобки с текстом → сохраняются; непарные/несовпадающие → вырезаются |
| 3 | `DateRule` | Даты → ISO `YYYY-MM-DD`; недостающие части → `00` |
| 4 | `LeadingZeroRule` | Однозначный числовой токен → с ведущим нулём |
| 5 | `SpaceToDashRule` | Пробелы → дефис; цепочки дефисов вокруг пробелов схлопываются в один, но намеренные дефисы без пробелов (`file--improved`) сохраняются |
| 6 | `TrimEdgeRule` | Обрезка не буквенно-цифровых символов по краям (ведущий `_` сохраняется и у файлов, и у папок; парная скобка на краю сохраняется) |
| 7 | `CaseRule` | Папки — с заглавной буквы, файлы — в нижнем регистре (кроме общепринятых имён вроде `README`, чей регистр сохраняется); у папок ведущий `_` сохраняется, капитализируется первая буква после него (`_private` → `_Private`) |

Порядок важен: `CaseRule` идёт последним (после схлопывания пробелов и обрезки кромок), что обеспечивает корректную капитализацию за один проход и идемпотентность.

## Публичное API

Импортируйте из пакета `normalizer`, а не из подмодулей напрямую:

```python
from pathlib import Path

from normalizer import build_normalizer, FilesystemNormalizer, load_excluder, load_includer

normalizer = build_normalizer()
normalizer.normalize("Отчёт 2020", is_dir=True)    # 'Otchiot_2020-00-00'

# Списки необязательны: load_excluder/load_includer вернут None, если файла нет.
excluder = load_excluder(Path("/path/to/project"))
includer = load_includer(Path("/path/to/project"))    # override exclude
fs = FilesystemNormalizer(build_normalizer(), excluder, includer)
renamed, skipped = fs.apply(Path("/path/to/dir"))
```

Экспортируется: `main`, `FilesystemNormalizer`, `PathExcluder`, `PathIncluder`, `load_excluder`, `load_includer`, `NameNormalizer`, `build_normalizer`, `Rule`, `TransliterationRule`, `BracketsRule`, `DateRule`, `LeadingZeroRule`, `CaseRule`, `SpaceToDashRule`, `TrimEdgeRule`.

Код проходит `ruff` и `mypy --strict` без замечаний; покрытие правил и сценариев ФС — в `tests/`.

### Windows: «ложные» изменения режима файлов

Скрипты-обёртки (`normalize.sh`, `normalize.command`) и `normalize_fs.py` хранятся в репозитории с битом исполняемости (`100755`), нужным на Linux/macOS. Файловая система Windows такой бит не хранит, поэтому `git status`/`git diff` на Windows постоянно показывают изменение режима (`old mode 100755 / new mode 100644`), которое не убирается даже через `git checkout --`.

Чтобы Git на Windows игнорировал режим файлов, выполните один раз в клоне:

```powershell
git config core.fileMode false
```

Это локальная настройка конкретного клона; содержимое файлов Git по-прежнему отслеживает, а Unix-пользователей она не затрагивает.
